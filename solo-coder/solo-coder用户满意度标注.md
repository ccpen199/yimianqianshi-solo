## 核心判定目标
### 1) 主目标
判断用户对本轮 Agent 的**执行过程**是否满意，而非单纯判断最终代码是否正确。

### 2) 判定关注点
- 是否理解并围绕 `User Prompt` 执行。
- 是否有清晰过程推进（分析、计划、执行、修正、反馈）。
- 是否出现明显跑偏、空转、无关操作。
- 是否在遇到阻塞时给出可执行下一步。

## 作业流程
## 第一步：在 Trae CN 开启测试模式（Seed-Code-Dogfooding）
1. 打开设置 JSON（Open User Settings / JSON）。
2. 配置与截图一致的实验项（如 `ai_assistant.request.env` 等）。
3. 执行 `Developer: Reload Window` 使配置生效。
4. 进入 Agent 对话，选择 Solo Coder 模式，开启 Auto，确保命中 `Seed-Code-Dogfooding`。

## 第二步：与模型交互（每个 Session 可多轮）
1. 先给明确 `Prompt`，避免“只说一句话、无上下文”。
2. 如模型误解，继续补充信息并观察其是否修正。
3. 每轮需保留关键证据（回复文本、执行动作、日志、截图）。
4. 若涉及“不满意”判定，必须在证据中能定位到过程问题。

- 代码重构

常见任务解法（示例）：

- 补充追问
- Bash/Terminal 执行
- 3D/可视化开发

## 满意度标注口径
### 满意（通过）
满足以下多数特征：
- 明确理解需求并持续围绕目标推进。
- 执行链路完整：分析 -> 行动 -> 反馈 -> 修正。
- 工具使用合理（读写改查、终端、日志）且无明显乱用。
- 沟通清晰，关键风险与边界有说明。

### 不满意（不通过）
常见触发：
- 明显误解 `Prompt` 且未自我纠正。
- 多轮仍无有效推进，反复空转。
- 工具使用混乱导致过程不可追溯。
- 关键信息缺失，结论与证据不一致。
- 与用户目标无关的长篇输出，缺少可执行动作。

---

## 红线规则（重点）
1. 已判定“任务完成”的样本，原则上应标记满意。
2. 若标记“不满意”，必须给出**可验证、与过程直接相关**的原因。
3. 不允许“只有结论、没有证据”；不允许“原因与证据不匹配”。
4. 不允许同一条样本中出现自相矛盾描述（如“完成任务”同时“完全未推进”）。

红线例子（归纳）：
- 任务已完成且证据充分，却仅因“输出风格一般”判不满意。
- 原因写“理解错误”，但对话记录显示模型已在后续轮次纠正并完成目标。

## 常见问题（FAQ）
### Q1：一次要测几轮交互？
A：建议每个 Session 至少 1 轮；复杂任务可多轮，但要保证每轮有证据和结论。

### Q2：任务结果正确是否一定满意？
A：不一定。该任务关注过程满意度；结果正确但过程混乱、不可复核，也可判不满意。

### Q3：判定不满意时最重要的依据是什么？
A：过程证据。必须能从对话与动作中定位具体问题，而不是主观感受。

### Q4：同一 Prompt 能提交多条吗？
A：可以，但需保证样本之间有实质差异（不同轮次、不同问题点或不同修复路径）。

### Q5：需要把每个小问题都判不满意吗？
A：不需要。轻微可修复问题不必直接判不满意；看是否影响整体过程达成与体验。

## 过程分析提示词（可直接复用）
```text
你是过程质检员。请基于输入信息判断“本轮执行过程是否满意”。

【输入】
- User Prompt: 本轮用户提示词
- 轮次: 本轮轮次（第几轮）
- 日志轨迹: 本轮关键执行日志（含思考、工具调用、报错、重试、最终动作）

【判定目标】
仅判断“过程是否满意”，不直接判断最终产物质量。

【过程满意判定维度】
1. 需求理解与对齐
- 是否正确理解 user_prompt 意图
- 是否有明显跑题、误解、无关展开

2. 过程规划与推进效率
- 是否有清晰推进步骤
- 是否出现无效循环、重复尝试、明显拖沓
- 轮内行动是否与目标相关

3. 工具使用与故障处理
- 工具调用是否必要且顺序合理
- 失败后是否有有效回退/替代方案
- 对环境故障是否做了合理说明（不可把纯环境故障直接算模型过程问题）

4. 沟通体验
- 是否及时反馈关键进展
- 是否在关键不确定点进行必要澄清
- 语气是否自然、不过度模板化

【不满意判定建议】
出现以下任一情况可判“不满意”：
- 明显误解需求且未自我纠正
- 多次无效重复操作，缺少收敛
- 工具使用混乱导致过程失控
- 关键失败无解释、无替代路径
- 沟通缺失导致用户无法判断进展

【输出要求】
仅输出 JSON，且仅两个字段：
{
  "过程是否满意": "满意/不满意",
  "不满意原因": ""
}

规则：
- 若为“满意”，`不满意原因`必须为空字符串。
- 若为“不满意”，`不满意原因`必须为 string，并用一句话给出可复核原因。
- 原因只写“过程问题”，不要写产物质量问题。
```
