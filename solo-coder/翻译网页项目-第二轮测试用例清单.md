# 翻译网页项目-第二轮测试用例清单（可直接用于标注/提交流程）

## 1. 第二轮验收标准（10分钟可定义）

- 翻译成功率：核心语言组合测试用例通过率 >= 95%
- 语言正确率：输出目标语言判定正确率 >= 98%
- 响应时间：P95 < 2s，P99 < 3s（在正常网络）
- 错误处理：断网/超时/接口失败 100% 有可读错误提示
- 可用性：桌面端（Chrome/Safari/Edge）和移动端（iOS Safari/Android Chrome）核心流程可用
- 防重复提交：请求进行中按钮禁用，连续点击不产生重复请求
- 稳定性：连续 100 次翻译无崩溃、无明显内存上涨、无页面卡死

## 3. 第二轮执行重点（按优先级）

### P0（必须先过）
- 功能正确性：翻译方向正确、目标语言不跑偏
- 错误处理：断网/超时/5xx 都有可理解提示
- 防重复提交：按钮禁用和请求去重有效

### P1（强烈建议）
- 稳定性：连续 50~100 次翻译无异常
- 兼容性：Chrome/Safari/Edge + 移动端浏览器核心可用
- 性能：P95 响应时间达标

### P2（体验优化）
- UI/交互：加载态、禁用态、空态、错误态文案清晰
- 可访问性：键盘可操作、基础语义标签完整

## 4. 自动化最低配（提交前必跑）

- 单元测试（输入校验）
  - 空输入、仅空格、超长输入、字符边界
- 接口 Mock 测试
  - 成功、超时、500、断网（fetch reject）
- E2E 冒烟
  - 打开页面 -> 输入文本 -> 点击翻译 -> 显示结果

建议提交门禁：
- 本地或 CI 必须通过：`unit + mock + e2e-smoke`
- 任一失败禁止合入

## 6. 缺陷分级与发布阻塞规则

- P0：核心功能不可用/错误结果严重/崩溃/无错误处理（阻塞发布）
- P1：主要流程可用但体验或稳定性明显问题（原则上修复后发布）
- P2：非核心体验问题（可排期修复）

发布判定建议：
- 存在未关闭 P0 -> 不可发布
- P1 > 3 个且含高频触发 -> 建议延期
- 仅 P2 -> 可带风险发布
