# Coding Agent 作业规范（试标）总结文档

## 1. 项目目标
本项目用于构建高质量训练数据，训练 Agent 对 System Prompt 中各类约束（Constraint）的准确遵循能力。
任务是：
- 针对给定编程任务编写约束（描述“如何做”，不是“做什么”）。
- 对约束进行分类与标签标注。
- 按单条与批量规则进行质检。

---

## 2. Constraint 核心定义
### 2.1 什么是 Constraint
Constraint 是对 Agent 行为方式的约束，不是具体任务本身。

- 错误示例（任务）：`帮我写一个 Python 脚本处理数据。`
- 正确示例（系统约束）：`修改文件前必须先用 Read 读取完整内容。`
- 正确示例（系统约束）：`所有新生成的 Python 代码必须包含 Type Hints。`

### 2.2 本次试标排除范围
- 不做多 agent/subagent 编排类约束。
- 不做 MCP Server 相关约束。
- 不做纯任务命令本身（如“实现一个登录页面”）。

结论：只能写“如何做”的约束，不能把“做什么”的任务指令当作约束。

---

## 3. 两个关键标签
每条约束都必须标注两个标签：

### 3.1 `task_dependent`（任务相关）
- `true`：约束依赖当前任务目标/上下文（如框架、路径、业务条件）。
- `false`：通用开发规范，与具体任务无关。

示例：
- `必须使用 FastAPI 框架` -> `task_dependent=true`
- `代码修改后必须运行测试` -> `task_dependent=false`

### 3.2 `framework_dependent`（框架相关）
- `true`：约束显式依赖 Claude Code 特有工具/能力（Read/Write/Edit/Bash/Glob/Grep）。
- `false`：通用工程规范，不依赖 Claude Code 特有工具。

示例：
- `修改文件前必须先用 Read 读取完整内容` -> `framework_dependent=true`
- `函数命名统一 snake_case` -> `framework_dependent=false`

---

## 4. 类别体系（Category）
每条约束需标注 1 个类别，类别如下：

- `C1 工具使用规范`：工具调用方式、顺序、格式、频率偏好。
- `C2 行为与流程约束`：执行步骤顺序、文件/目录操作限制、单次修改范围。
- `C3 代码规范`：代码风格、命名约定、注释/文档要求。
- `C4 实现偏好`：技术选型、错误处理方式、性能要求。
- `C5 架构约束`：代码组织、目录结构、设计模式、分层。
- `C6 安全策略`：安全边界、禁止操作、敏感数据处理。
- `C7 测试要求`：测试框架选择、测试编写规范、覆盖率。
- `C8 构建与版本控制`：包管理器、构建命令、Git 操作规范。
- `C9 输出与回复`：Agent 回复格式、语言、风格、详细程度。
- `C10 其他`：无法归入以上类别但合理的约束。

---

## 5. 作业编写要求（10 条约束）
## 5.1 数量与覆盖
- 必须写 **10 条** constraint。
- 类别覆盖至少 **6 类**。
- 同一组合（类别 + `task_dependent` + `framework_dependent`）重复数量不得超过 **3 条**。

## 5.2 前后分布规则
前 8 条（第 1-8 条）要求：
- 必须全部 `task_dependent=true`。
- 其中 `framework_dependent=true` 至少 4 条。
- 其中 `framework_dependent=false` 至多 4 条。

后 2 条（第 9-10 条）要求：
- 必须全部 `task_dependent=false`。
- 其中 `framework_dependent=true` 至少 1 条。
- 其中 `framework_dependent=false` 至多 1 条。

## 5.3 数据质量要求
- 多条数据不能模板化重复，每条必须不同。
- 每条约束必须可验证、不可含糊、无歧义。

---

## 6. Claude Code 能力边界（用于可执行性判断）
评估约束可执行性时，以以下能力为准：
- 核心工具：`Read`、`Write`、`Edit`、`Bash`、`Glob`、`Grep`。
- 可通过 Bash 运行命令、脚本、编译流程、测试。
- 可基于命令输出与文件内容进行判断。

限制：
- 不能依赖图形界面手动操作。
- 不能要求必须真人交互的手工验证。

---

## 7. 单条约束质检维度（12 项）
单条质检要求“全通过才通过”。

### 7.1 质量维度（1-7）
1. **信息充分性**：是否明确“何时做、做什么、做到什么程度”。
2. **约束必要性**：是否真正限制行为、避免错误，非常识空话。
3. **表达精确性**：是否可量化、可判定，避免“清晰/合理/尽量”等模糊词。
4. **结果可验证性**：是否能通过工具输出/文件内容客观验证。
5. **工具使用准确性**：涉及工具时是否符合真实能力。
6. **求解方案独立性**：是否为新增约束而非重复用户原始要求。
7. **场景匹配性**：是否符合真实任务场景，不引入不存在前提。

### 7.2 标签与类别维度（8-12）
8. **`task_dependent` 标签正确性**。
9. **`framework_dependent` 标签正确性**。
10. **`category` 分类正确性**。
11. **标签理由完整性**：两个标签都必须有理由。
12. **标签理由合理性**：理由与约束语义一致，不自相矛盾。

---

## 9. 批量质检规则
批量质检是对整批约束做“硬门槛检查”，任一项失败则整批失败。

## 9.1 基础指标（必须全过）
1. **约束数量检查**：必须恰好 10 条。
2. **类别覆盖检查**：类别数必须 >= 6。
3. **组合重复性检查**：任一（类别 + `task_dependent` + `framework_dependent`）组合不得 > 3 条。

## 9.2 约束分布（必须全过）
1. **前 8 条分布**：全部 `task_dependent=true`，且 `framework_dependent=true` >=4、`framework_dependent=false` <=4。
2. **后 2 条分布**：全部 `task_dependent=false`，且 `framework_dependent=true` >=1、`framework_dependent=false` <=1。

## 9.3 批次级质量（必须全过）
1. **约束间一致性**：无直接冲突；有潜在冲突需边界条件化解。
2. **内容无重复性**：不能核心规则同义改写、仅换措辞。

---

## 10. 批量质检输出格式
### 10.1 通过
```text
整体质检结果： TRUE
```

### 10.2 不通过
```text
整体质检结果： FALSE

不通过项：
1. [校验类别 - 具体检查项]：[具体原因说明]
2. [校验类别 - 具体检查项]：[具体原因说明]
...
```

## 11. 常见 Badcase（汇总）
### 11.1 单条常见问题
- 约束必要性不足：仅是常识或默认行为，去掉不影响结果。
- 信息充分性不足：没有给出执行边界、对象或程度。
- 结果不可验证：使用主观描述，无法客观检查。
- 场景不匹配：假设了任务中不存在的环境/前提。
- 标签错配：内容涉及工具却标 `framework_dependent=false`；或任务特定却标 `task_dependent=false`。
- 标签理由不完整/不合理：缺失理由、占位词理由、理由与标签含义冲突。

### 11.2 批量常见问题
- 数量不是 10 条。
- 类别覆盖不足 6 类。
- 某组合出现 4 条及以上。
- 前 8 条混入 `task_dependent=false`。
- 前 8 条 `framework_dependent=true` 少于 4。
- 后 2 条混入 `task_dependent=true`。
- 后 2 条 `framework_dependent=true` 为 0。
- 批内存在冲突约束（如“全部用 Write 新建”与“全部用 Edit 修改”且无条件分流）。
- 批内核心规则重复（如“加 type hints”与“函数加类型注解”本质同义）。
