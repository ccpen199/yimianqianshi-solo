# Coding Agent 约束质量检验标准（试标）总结 - 第二份

## 1. 文档目的
本标准用于评估一批约束（constraints）的质量是否达标，核心关注：
- 单条约束是否写得清晰、必要、可执行、可验证。
- 一整批约束之间是否一致、无冲突、无重复。

---

## 2. 单条约束质检维度（7项）
单条约束需从以下 7 个维度检查：

## 2.1 信息充分性
核心问题：约束是否明确说明“何时做、做什么、做到什么程度”？
- 通过标准：对象、触发时机、执行动作、结果边界完整。
- 常见不通过：只写原则，不给执行条件或范围。

## 2.2 约束必要性
核心问题：去掉该约束后，Agent 输出是否会实质变化？
- 通过标准：约束能限制行为并减少错误/偏差。
- 常见不通过：常识性废话或默认会做的事情（无约束增益）。

## 2.3 表达精确性
核心问题：约束中的关键元素是否具体、可判定、可落地？
- 通过标准：尽量可量化（路径、阈值、次数、格式、工具名）。
- 常见不通过：出现“合理/清晰/尽量/适当”等模糊词且无补充标准。

## 2.4 结果可验证性
核心问题：审核者能否仅依据文件内容或命令输出客观判定是否遵守？
- 通过标准：可通过 Read/Grep/Bash 输出进行核验。
- 常见不通过：依赖主观判断或不可复现的人为观察。

## 2.5 工具使用准确性
核心问题：约束中工具用法是否符合 Claude Code 的真实能力？
- 通过标准：Read/Write/Edit/Bash/Glob/Grep 使用场景与参数合理。
- 常见不通过：要求工具执行其能力边界外动作。

## 2.6 求解方案独立性
核心问题：该约束是否为额外有效约束，而非复述任务本身？
- 通过标准：约束补充了执行策略/质量门槛，而不是重复用户目标。
- 常见不通过：把“要做什么”直接重写成“约束”。

## 2.7 场景匹配性
核心问题：约束前提是否与任务场景一致，不引入不存在条件？
- 通过标准：与给定项目上下文一致，边界条件可解释。
- 常见不通过：预设任务中不存在的技术栈、文件、流程或环境。

---

## 3. 批次级质检维度（2项）
批量评估时，除单条质量外，还要检查整批一致性。

## 3.1 约束间一致性
核心问题：批次内多条约束是否可同时满足？
- 通过标准：约束之间无直接冲突；若存在潜在冲突，给出清晰分流条件。
- 不通过示例：A 要求“所有修改优先 Write 新建”，B 要求“所有修改优先 Edit 直接改”，且没有条件边界。

## 3.2 内容无重复性
核心问题：是否存在核心规则重复，仅换措辞？
- 通过标准：每条约束都针对不同风险点、对象或阶段。
- 不通过示例：
  - “所有 Python 代码必须加 Type Hints”
  - “编写 Python 函数时必须添加类型注解”
  两条核心语义相同，属于重复。

## 4. Good Case / Badcase 评审要点
截图中的样例体现了以下判定口径：

- Good case 特征：
  - 约束内容可执行、可验证。
  - 标签（如 category、task_dependent、framework_dependent）与内容一致。
  - 对异常处理、流程顺序、工具选择给出可落地要求。

- Badcase 常见问题：
  - 约束含糊，无法转成明确动作。
  - 约束复述任务需求，缺乏新增约束价值。
  - 标签与内容不一致（例如明明依赖工具却标记为框架无关）。
  - 要求与任务上下文不匹配，或引入不存在前提。

---

## 5. 实操检查清单
用于快速审稿：
- [ ] 每条约束都能回答“何时、做什么、做到什么程度”。
- [ ] 去掉该约束后输出会发生明显变化（证明必要性）。
- [ ] 约束无模糊词，或模糊词已被量化定义。
- [ ] 审核可仅依赖命令输出/文件证据完成。
- [ ] 工具使用符合 Claude Code 实际能力边界。
- [ ] 约束不是任务重述，而是策略或质量约束。
- [ ] 与任务场景一致，不预设不存在条件。
- [ ] 批次内无冲突约束。
- [ ] 批次内无核心语义重复约束。

---

## 6. 建议输出模板（评审结论）
```text
质检结果：TRUE/FALSE

若 FALSE：
不通过项：
1. [维度名称]：[具体原因]
2. [维度名称]：[具体原因]

补充说明：
- 单条问题：...
- 批次问题：...
- 修复建议：...
```

---

## 7. 一句话总结
第二份文档的核心是：先用 7 个维度保证“单条约束质量”，再用“约束间一致性 + 内容无重复性”保证“整批可用性”。
