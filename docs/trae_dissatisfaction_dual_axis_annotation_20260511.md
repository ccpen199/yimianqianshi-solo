## 使用资料

1. 官方标注指导：`官方文档指导.log`
   - 不满意理由不能只写“无法运行、页面打不开、接口失败”等笼统结论。
   - 至少要覆盖范围/对象、现象/证据、与需求偏差、影响范围、复现条件、严重程度中的两项。

2. 会话轮次缓存：

   - `docs/data/generated/trae_session_rounds/may-992.json`

   - `docs/data/generated/trae_session_rounds/may-994.json`

3. 每轮可用字段：
  - `conversation`：本轮验收输入或修复要求。
   - `logTrace`：本轮模型规划、工具调用、启动命令、测试命令、最终总结。
  - `screenshots`：该轮验收输入中附带的截图证据。
   - `sessionId/sessionComposite`：用于确认轮次顺序，不作为不满意原因正文。

## 行对齐规则

不满意原因列描述的是“当前轮对应产物/修复任务为什么仍未达到验收要求”。

最高优先级是先锁定第 N 行的评价对象。评价对象必须来自第 N 行 `conversation` 本身：本轮要求修复哪个页面、入口、按钮、接口、角色路径、业务规则，就评价这些对象是否达标。第 N+1 行只能帮助判断“第 N 行同一个问题是否仍然没修好”，不能把第 N+1 行新出现的问题挪到第 N 行。

因此第 N 行的标注应先使用：

```text
第 N 行 conversation 中的任务、反馈或修复要求
+ 第 N 行 logTrace 中的执行过程、测试方式和交付总结
+ 第 N 行 screenshots 中的截图证据
```

第 N+1 行 `conversation` 只能作为辅助证据，不能替代第 N 行的评价对象。只有当第 N+1 行明确是在复验第 N 行同一个问题时，才可以用它补充“上一轮没有修好”的结论；如果第 N+1 行已经切换到新问题，不能把新问题写进第 N 行的产物不满意。

第一轮通常是完整主需求，不是验收反馈。第一轮可以用第 2 轮的验收输入判断初版产物问题，但从第二轮开始，必须优先评价当前轮本身要求处理的问题。

最后一行不返回 `-`，也不写证据缺失式话术。必须直接基于当前轮已有信息完成总结：

```text
产物不满意：结合当前轮输入、日志轨迹、截图和交付总结说明当前仍未闭环的页面、接口、数据或业务链路问题。
过程不满意：结合当前轮执行记录说明验收是否充分，若不充分要明确缺少启动证据、端口归属、页面点击、核心 API、异常场景或截图证据。
```

## 输出格式

每一行必须使用双轴结构：

```text
产物不满意：说明当前轮任务对应的页面/接口/模块/功能的具体问题、范围对象、证据、与需求偏差和影响。
过程不满意：说明模型执行过程中的问题，例如未启动、未验收、只跑 curl 未看页面、反复导航、日志缺失、未按端口安全处理、未按需求逐项验证等。
```

可以写成单行，便于飞书表格粘贴：

```text
产物不满意：...。过程不满意：...。
```

每个单元格只能有一组双轴结论，只能出现一次 `产物不满意：` 和一次 `过程不满意：`。如果模型返回同一段双轴内容两遍，应直接去重，只保留第一组完整结论；不能为了显得更详细而重复粘贴相同句子。

## 判断方法

1. 先锁定当前轮评价对象。
   - 评价对象来自当前行 `conversation`，必须写清本轮正在处理的页面、模块、入口、按钮、接口、角色路径或业务规则。
   - 如果当前轮是“首页内容简单、登录后跳转逻辑错误”，产物不满意就必须评价首页内容、展示端信息架构、登录后路由和前后台角色隔离。
   - 不能因为下一轮出现“前后端未对接”“接口只返回成功信息”等新问题，就把当前轮改写成前后端未对接。

2. 再读主需求。
   - 第一轮 `conversation` 通常包含完整业务需求和端口/验收/安全约束。
   - 后续轮次的验收输入必须回到主需求判断偏差，不能只复述一句“页面有问题”。

3. 再读后续验收输入。
   - 这是辅助证据，不是无条件的直接来源。
   - 如果后续验收输入仍在说当前轮同一个问题，例如“上一个问题没有解决”“仍然失败”“点击同一按钮还是无响应”，可以用来判断当前轮修复未闭环。
   - 如果后续验收输入已经切换到新问题，例如当前轮是“首页内容简单、登录后跳转错误”，后续轮变成“前端没有接到后端”，不能把后续的新问题写成当前轮的产物不满意。
   - 需要保留具体对象和现象，例如“充值中心和天猫超市入口空白”“热门榜/推荐榜/垂直榜共用同一批数据”。

4. 再读上一轮日志轨迹。
   - 看是否实际创建/修改了对应文件。
   - 看是否启动了前后端。
   - 看是否只做 API curl，还是也做了浏览器页面验收。
   - 看是否出现反复试错、长时间无收敛、日志缺失、端口处理不安全等过程问题。

5. 最后看截图。
   - 截图用于增强证据，而不是替代分析。
   - 例如黑屏、`ERR_CONNECTION_REFUSED`、broken image、空白页面、榜单数据重复，都要转写成具体文字。

## 质量自检

生成每条不满意原因后必须自检两件事：

1. 是否评价了当前轮任务。
   - 不合格：当前轮是“首页内容简单、登录后跳转错误”，结果只写“前端页面没有接到后端”。
   - 合格：明确写首页展示端内容单薄、登录后跳入管理后台、前台浏览和后台治理路径混乱。

2. 是否足够具体。
   - 不合格：只写“页面没有渲染结果”“跳转逻辑错误”“权限错误”。
   - 合格：写清哪个页面/入口/角色路径、出现什么现象、影响哪些核心流程，例如“首页展示端缺少资讯、论坛、博客、产品内容和推荐信息，登录后不能回到消费者前台，导致未登录浏览、登录后继续浏览和后台管理角色边界都无法验收”。

3. 是否重复输出。
   - 不合格：`产物不满意：词条管理页面没有完成开发...。过程不满意：处理过程仍围绕管理后台点击循环...。产物不满意：词条管理页面没有完成开发...。过程不满意：处理过程仍围绕管理后台点击循环...。`
   - 合格：同一条只保留一次完整双轴结论；重复内容不是更充分的分析，而是粘贴污染。

## 本次标注原则

- 只改 `dissatisfactionReason`、`annotationVersion`、`annotationUpdatedAt`、`annotationSource`。
- 不改 Session、会话、序号、截图、日志轨迹等列。
- 不把产物问题压缩成短标签，例如不要写“页面黑屏”，要写清楚访问地址、影响模块和主流程。
- 不允许反馈偏移：产物不满意必须评价当前轮任务本身，不得只写下一轮的新问题。
- 产物不满意至少写清两个要素：范围/对象（哪个页面、模块、按钮、接口、角色路径）和影响范围（影响哪些核心流程、业务目标或验收链路）。
- 当前轮评价对象缺失时，即使文字较长也判为不合格；范围/对象和影响范围缺失时，即使方向正确也判为不合格。
- 一行不得重复输出相同的 `产物不满意/过程不满意` 组合；重复时只保留第一组完整结论。
- 即使产物问题很明确，也必须继续分析过程是否存在问题。
- 如果过程没有明显失败，也要写明“过程不满意”的判断依据，例如验收不充分、缺少浏览器截图、只验证接口、启动说明不完整。
- 禁止输出证据缺失式、等待后续输入式、引用提出方要求式的话术。
- 不以提出方作为叙述主体；需要表达要求来源时写“业务要求”“验收要求”“主需求”“页面表现”或“交付过程”。
- 不能因为缺少后一轮验收输入就写证据缺失句，必须从当前轮的 prompt、日志轨迹、截图、启动和验收记录中直接提炼问题。

## 示例

原始验收输入：

```text
页面黑屏，过程方面，从会员体系到消息中心，都没有垂直业务作出正确的组件选择。
没有引入异步数据处理：Vue Query (TanStack Query)。
```

合格标注：

```text
产物不满意：前端 http://localhost:9911 首屏黑屏，会员体系、消息中心等电商垂直模块没有形成可用页面，核心购物路径无法进入。过程不满意：日志虽然创建了大量模块并跑了 API/启动命令，但验收偏后端 curl 和完成清单，未基于浏览器截图确认渲染，也未按要求建立稳定的异步数据状态处理。
```

反例：

```text
产物不满意：前端页面没有接到后端，导致只返回后端的接口成功信息。没有页面渲染结果。
```

问题：

- 评价对象偏移：当前轮任务是“首页内容简单、登录后跳转逻辑错误”，却只写了下一轮“前后端未对接”的新问题。
- 详细度不足：没有说明是哪个页面、哪个入口、影响了展示端浏览、登录后返回前台、后台管理隔离等哪些核心流程。
- 缺少过程轴：只写产物现象，没有继续判断修复过程是否围绕当前轮对象做页面验收、路由断言和角色隔离回归。

合格写法：

```text
产物不满意：首页内容仍然只有简单标题、顶部内容和分类产品名称，产品吧展示端缺少资讯、论坛、博客、产品内容和推荐信息；登录后还从消费者展示端错误跳入管理后台，前台浏览、登录后继续浏览和后台治理角色路径被混在一起。过程不满意：修复过程没有围绕首页信息架构、登录后路由、角色权限和前后台隔离做端到端验证，也缺少从未登录浏览到登录后返回前台的页面截图和路由断言。
```

重复输出反例：

```text
产物不满意：词条管理页面没有完成开发，页面为空白，产品吧 PRD 中名词/词条沉淀、管理和维护能力缺失。过程不满意：处理过程仍围绕管理后台点击循环和服务重启，缺少对词条页面组件、路由挂载、接口数据、空状态渲染和页面刷新结果的直接验证。产物不满意：词条管理页面没有完成开发，页面为空白，产品吧 PRD 中名词/词条沉淀、管理和维护能力缺失。过程不满意：处理过程仍围绕管理后台点击循环和服务重启，缺少对词条页面组件、路由挂载、接口数据、空状态渲染和页面刷新结果的直接验证。
```

合格写法：


## 后续 LLM-Agent 建议流程

2026-05-12 已落地到 Workbench：

```text
批量项目组 -> LLM-Agent -> 选择 DeepSeek V4 Pro -> 生成当前组
POST /api/annotate-dissatisfaction
```

服务端只写回本机 `data/generated/trae_session_rounds/{order}.json` 的不满意列相关字段，不改会话、截图和日志轨迹。没有 `DEEPSEEK_API_KEY` 时按钮会显示不可用，避免误以为已经走了大模型。

1. 输入一个批量组名或序号列表。
2. 读取 `prompt_state.json` 找到组内序号。
3. 逐个读取 `trae_session_rounds/{order}.json`。
4. 对每个 order 内的 rows 按时间顺序处理。
5. 对第 N 行构造上下文：
   - `mainRequirement = rows[0].conversation`
   - `currentConversation = rows[N].conversation`
   - `currentTrace = rows[N].logTrace`
   - `currentScreenshots = rows[N].screenshots`
   - `nextValidationInput = rows[N+1].conversation`，只作为同一问题复验辅助证据
   - `nextScreenshots = rows[N+1].screenshots`，只作为同一问题复验辅助证据
6. 输出严格 JSON：

```json
{
  "items": [
    {
      "order": "may-991",
      "round": 1,
      "dissatisfactionReason": "产物不满意：...。过程不满意：...。"
    }
  ]
}
```

7. 写回缓存时只更新不满意列相关字段，并保留原始轮次、日志轨迹、截图和会话 ID。
