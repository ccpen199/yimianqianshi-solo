## 使用资料

1. 官方标注指导：`官方文档指导.log`
   - 不满意理由不能只写“无法运行、页面打不开、接口失败”等笼统结论。
   - 至少要覆盖范围/对象、现象/证据、与需求偏差、影响范围、复现条件、严重程度中的两项。

2. 会话轮次缓存：
   - `docs/data/generated/trae_session_rounds/may-991.json`
   - `docs/data/generated/trae_session_rounds/may-992.json`
   - `docs/data/generated/trae_session_rounds/may-993.json`
   - `docs/data/generated/trae_session_rounds/may-994.json`

3. 每轮可用字段：
  - `conversation`：本轮任务输入、修复要求或验收反馈。
   - `logTrace`：本轮模型规划、工具调用、启动命令、测试命令、最终总结。
  - `screenshots`：原始会话中该轮输入附带的截图；用于不满意列时，第 N 行应使用第 N+1 行截图作为第 N 轮产物结果截图。
   - `sessionId/sessionComposite`：用于确认轮次顺序，不作为不满意原因正文。

## 行对齐规则

不满意原因列描述的是“第 N 轮产出的结果为什么没有达到验收要求”。

最高优先级是先锁定第 N 行代表的是“第 N 轮产出”，不是“第 N 行文字本身”。第 N+1 行 `conversation` 通常是对第 N 轮产物的验收反馈，因此它是第 N 行产物不满意的主要证据来源；第 N 行 `conversation` 用来判断第 N 轮当时做了什么任务、修了哪个对象，防止反馈内容脱离本轮交付背景。

因此第 N 行的标注应使用：

```text
产物轴 = 第 N+1 行 conversation 中对第 N 轮结果的反馈
       + 第 N+1 行 screenshots 前移后的结果截图
       + 第 N 行 conversation 对本轮任务背景的限定
过程轴 = 第 N 行 logTrace 中的执行过程、测试方式和交付总结
```

不能把第 N+1 行反馈理解成“下一轮自己的产物问题”。在批量会话表里，第 N+1 行反馈是查看第 N 轮结果后发出的验收意见，必须前移到第 N 行产物不满意里。第 N 行日志轨迹仍然留在第 N 行，用于判断第 N 轮过程是否充分。

第一轮通常是完整主需求，不是验收反馈。第一轮产物不满意应使用第 2 轮反馈评价初版产物；第二轮以后同理，第 N 行使用第 N+1 行反馈评价第 N 轮修复后的产物。最后一行没有后续反馈时，才退回当前行 `conversation`、当前行 `logTrace` 和当前行/已前移截图做保守总结。

最后一行不返回 `-`，也不写证据缺失式话术。必须直接基于当前轮已有信息完成总结：

```text
产物不满意：结合当前轮输入、日志轨迹、截图和交付总结说明当前仍未闭环的页面、接口、数据或业务链路问题。
过程不满意：结合当前轮执行记录说明验收是否充分，若不充分要明确缺少启动证据、端口归属、页面点击、核心 API、异常场景或截图证据。
```

## 输出格式

每一行必须使用双轴结构：

```text
产物不满意：说明当前轮任务对应的页面/接口/模块/功能的确定性问题、范围对象、证据和与需求的偏差。
过程不满意：说明模型执行过程中的问题，例如未启动、未验收、只跑 curl 未看页面、反复导航、日志缺失、未按端口安全处理、未按需求逐项验证等。
```

可以写成单行，便于飞书表格粘贴：

```text
产物不满意：...。过程不满意：...。
```

每个单元格只能有一组双轴结论，只能出现一次 `产物不满意：` 和一次 `过程不满意：`。如果模型返回同一段双轴内容两遍，应直接去重，只保留第一组完整结论；不能为了显得更详细而重复粘贴相同句子。

## 判断方法

1. 先锁定当前轮评价对象。
   - 评价对象来自第 N 轮产物结果，主要由第 N+1 行反馈和前移截图证明；必须写清本轮产物暴露出的页面、模块、入口、按钮、接口、角色路径或业务规则问题。
   - 第 N 行 `conversation` 用于理解第 N 轮原本要修什么、做什么，不能替代第 N+1 行反馈。
   - 如果第 N+1 行反馈是“首页内容简单、登录后跳转逻辑错误”，第 N 行产物不满意就必须评价首页内容、展示端信息架构、登录后路由和前后台角色隔离。

2. 再读主需求。
   - 第一轮 `conversation` 通常包含完整业务需求和端口/验收/安全约束。
   - 后续轮次的验收输入必须回到主需求判断偏差，不能只复述一句“页面有问题”。

3. 再读后续验收输入。
   - 这是第 N 轮产物问题的主要证据，不是辅助证据。
   - 如果第 N+1 行反馈说“上一个问题没有解决”“仍然失败”“点击同一按钮还是无响应”，第 N 行产物轴要写清同一问题仍未修好。
   - 如果第 N+1 行反馈切换到新的页面问题，例如第 N 行任务是修首页数据，第 N+1 行反馈变成“标签内容全都一样”，也应写到第 N 行，因为这说明第 N 轮交付后的产物暴露出分类筛选问题。
   - 需要保留具体对象和现象，例如“充值中心和天猫超市入口空白”“热门榜/推荐榜/垂直榜共用同一批数据”。

4. 再读当前行日志轨迹。
   - 看是否实际创建/修改了对应文件。
   - 看是否启动了前后端。
   - 看是否只做 API curl，还是也做了浏览器页面验收。
   - 看是否出现反复试错、长时间无收敛、日志缺失、端口处理不安全等过程问题。

5. 最后看截图。
   - 截图用于增强证据，而不是替代分析。
   - 第 N 行截图列已经展示第 N+1 行会话中的图片，含义是第 N 轮结果截图；写第 N 行不满意原因时应直接使用这张前移后的截图。
   - 例如黑屏、`ERR_CONNECTION_REFUSED`、broken image、空白页面、榜单数据重复，都要转写成具体文字。

## 质量自检

生成每条不满意原因后必须自检两件事：

1. 是否评价了第 N 轮产物。
   - 不合格：第 N+1 行反馈已经说明“标签内容全都一样，没有进行分类处理”，结果仍只写第 N 行原始任务里的“首页没有数据”。
   - 合格：明确写第 N 轮产物在产品设计、用户研究、数据分析、运营增长等标签下展示内容完全相同，分类筛选没有生效。

2. 是否足够具体。
   - 不合格：只写“页面没有渲染结果”“跳转逻辑错误”“权限错误”。
   - 合格：写清哪个页面/入口/角色路径、出现什么现象、影响哪些核心流程，例如“首页展示端缺少资讯、论坛、博客、产品内容和推荐信息，登录后不能回到消费者前台，导致未登录浏览、登录后继续浏览和后台管理角色边界都无法验收”。

3. 是否重复输出。
   - 不合格：`产物不满意：词条管理页面没有完成开发...。过程不满意：处理过程仍围绕管理后台点击循环...。产物不满意：词条管理页面没有完成开发...。过程不满意：处理过程仍围绕管理后台点击循环...。`
   - 合格：同一条只保留一次完整双轴结论；重复内容不是更充分的分析，而是粘贴污染。

4. 是否把反馈留在错误行。
   - 不合格：第 N+1 行反馈是“普通用户登录不成功，没有任何响应信息”，却把第 N 行仍写成“页面链路存在缺陷没有内容”，并把登录问题留给第 N+1 行。
   - 不合格：第 N+1 行反馈是“搜索功能存在问题”，第 N 行却只复述第 N 行修复要求，没有评价搜索入口、搜索结果或搜索接口。
   - 合格：产物不满意必须吸收第 N+1 行反馈中已经明确出现的页面、接口、按钮、数据或业务规则问题，并前移到第 N 行。

## 本次标注原则

- 只改 `dissatisfactionReason`、`annotationVersion`、`annotationUpdatedAt`、`annotationSource`。
- 不改 Session、会话、序号、截图、日志轨迹等列。
- 不把产物问题压缩成短标签，例如不要写“页面黑屏”，要写清楚访问地址、影响模块和主流程。
- 不允许反馈错位：第 N+1 行反馈要前移评价第 N 轮产物，不得滞留在第 N+1 行；第 N 行日志轨迹不得错用于第 N+1 行过程分析。
- 产物不满意至少写清两个要素：范围/对象（哪个页面、模块、按钮、接口、角色路径）和影响范围（影响哪些核心流程、业务目标或验收链路）。
- 产物判断必须有确定性证据：只能写第 N+1 行反馈、前移截图、第 N 行日志轨迹或第 N 行任务背景中已经明确出现的页面事实、接口事实、数据事实或业务规则偏差；禁止把推测性影响、主观判断成本、可能造成的理解偏差写成产物问题。
- 时间显示类问题只能写确定事实，例如“相对时间字段把一分钟前显示为三小时前”；没有证据证明排序、提醒、审计、内容流受到影响时，不能写“影响对内容时间线的判断”。
- 第 N+1 行反馈对象缺失时，即使文字较长也判为不合格；范围/对象和影响范围缺失时，即使方向正确也判为不合格。
- 一行不得重复输出相同的 `产物不满意/过程不满意` 组合；重复时只保留第一组完整结论。
- 过程不满意必须围绕第 N 行日志轨迹里的执行过程，不默认写“上一轮交付缺少...”；产物轴可以来自第 N+1 行反馈，但过程轴不能跟着右移。
- 过程不满意不能同质化套模板；禁止连续多行复用“缺少复现路径、文件修改、接口请求、浏览器操作、截图对照和回归结果”“问题没有被完整闭环验证”“当前轮需要把修复文件、启动状态、复现路径和验收结果串起来”等通用句。
- 过程不满意每行至少绑定一个当前轮特征：例如匹配列表、刷新按钮、白屏、路由挂载、登录接口、分类标签、右滑按钮、个人信息页、启动命令、curl、截图或具体 `filePath`，并说明这些证据链哪里断了。
- 即使产物问题很明确，也必须继续分析过程是否存在问题。
- 如果过程没有明显失败，也要写明“过程不满意”的判断依据，例如验收不充分、缺少浏览器截图、只验证接口、启动说明不完整。
- 禁止输出证据缺失式、等待后续输入式、引用提出方要求式的话术。
- 不以提出方作为叙述主体；需要表达要求来源时写“业务要求”“验收要求”“主需求”“页面表现”或“交付过程”。
- 不能因为缺少后一轮验收输入就写证据缺失句，必须从当前轮的 prompt、日志轨迹、截图、启动和验收记录中直接提炼问题。

## 示例

原始验收输入：

```text
页面黑屏，过程方面，从会员体系到消息中心，都没有垂直业务作出正确的组件选择。
没有引入异步数据处理：Vue Query (TanStack Query)。
```

合格标注：

```text
产物不满意：前端 http://localhost:9911 首屏黑屏，会员体系、消息中心等电商垂直模块没有形成可用页面，核心购物路径无法进入。过程不满意：日志虽然创建了大量模块并跑了 API/启动命令，但验收偏后端 curl 和完成清单，未基于浏览器截图确认渲染，也未按要求建立稳定的异步数据状态处理。
```

错位反例：

```text
第 N 行任务：修复首页没有数据的问题。
第 N+1 行反馈：全部产品设计、用户研究、数据分析、运营增长、职业发展、技术相关、最新、热门这些标签内容全是一样的，没有进行分类处理。
错误标注：产物不满意：首页没有数据，各个模块没有数据。过程不满意：当前轮修复和验收过程缺少围绕该问题的复现路径、文件修改、接口请求、浏览器操作、截图对照和回归结果。
```

问题：

- 产物轴没有使用第 N+1 行验收反馈，仍停留在第 N 行修复前的问题。
- 详细度不足：没有说明哪些标签内容相同、分类筛选如何失效、影响导航浏览和内容发现链路。
- 过程轴套模板，没有结合第 N 行日志判断是否验证了分类点击、查询参数、后端筛选结果和页面列表差异。

合格写法：

```text
产物不满意：产品设计、用户研究、数据分析、运营增长、职业发展、技术相关、最新、热门等标签切换后展示内容完全相同，首页分类筛选没有生效，内容发现和主题导航链路无法验收。过程不满意：第 N 轮日志需要证明标签点击、查询参数、后端筛选结果和页面列表差异已经逐项回归，不能只用补充首页数据替代分类逻辑验证。
```

轮次前移反例：

```text
第 N 行任务：没有遵守业务规则，导致页面链路存在缺陷没有内容。
第 N+1 行反馈：普通用户登录不成功，没有任何响应信息。
错误标注：产物不满意：页面链路存在缺陷没有内容。过程不满意：当前轮交付缺少围绕该问题的复现路径、文件修改、接口请求、浏览器操作、截图对照和回归结果。
```

问题：

- 产物轴没有把第 N+1 行登录失败反馈前移到第 N 行。
- 过程轴泛化：过程不满意只套用了“缺少复现路径...”模板，没有说明登录表单提交、认证接口返回、错误提示和成功跳转是否被验证。

合格写法：

```text
产物不满意：普通用户登录提交后没有成功，也没有任何响应信息或错误提示，认证入口无法完成登录闭环，普通用户后续浏览、互动和个人功能都无法进入。过程不满意：第 N 轮日志需要覆盖登录表单提交、认证接口返回、错误提示渲染和成功跳转验证；如果只修页面链路或数据内容，登录链路是否可用仍无法被证明。
```

过程同质化反例：

```text
产物不满意：页面几乎没开放，只是显示了一行字暂无匹配和刷新按钮。过程不满意：上一轮交付缺少围绕该问题的复现路径、文件修改、接口请求、浏览器操作、截图对照和回归结果，问题没有被完整闭环验证。
产物不满意：页面导入了，已经有网页标签但是渲染是白色的。过程不满意：上一轮交付缺少围绕该问题的复现路径、文件修改、接口请求、浏览器操作、截图对照和回归结果，问题没有被完整闭环验证。
```

合格写法：

```text
产物不满意：页面几乎没开放，只是显示了一行字暂无匹配用户和刷新按钮。过程不满意：过程记录没有围绕匹配列表数据、空状态文案、刷新按钮和随机匹配接口做逐项回归，难以确认“暂无匹配”是否被真实数据链路修复。
产物不满意：页面导入了，已经有网页标签但是渲染是白色的，说明前端的渲染路径还存在问题。过程不满意：过程缺少对前端入口、路由挂载、控制台报错和首屏 DOM 渲染结果的连续核验，不能只通过文件修改判断白屏问题已消除。
```

## 后续 LLM-Agent 建议流程

1. 输入一个批量组名或序号列表。
2. 读取 `prompt_state.json` 找到组内序号。
3. 逐个读取 `trae_session_rounds/{order}.json`。
4. 对每个 order 内的 rows 按时间顺序处理。
5. 对第 N 行构造上下文：
   - `mainRequirement = rows[0].conversation`
   - `currentTaskConversation = rows[N].conversation`
   - `currentTrace = rows[N].logTrace`
   - `validationFeedback = rows[N+1].conversation`，优先作为第 N 轮产物问题来源
   - `resultScreenshots = rows[N+1].screenshots`，前移作为第 N 轮结果截图
   - 如果没有 `rows[N+1]`，再退回 `rows[N].conversation`、`rows[N].logTrace` 和现有截图做最终总结
6. 输出严格 JSON：

