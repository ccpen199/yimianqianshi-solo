# PRD：倒推网易云音乐-私藏推荐产品需求文档

网易云音乐在近期更新的5.9.1版本中，新推出了私藏推荐功能。本文将从私藏推荐PM
的角度，倒推这个功能的产品需求文档。
网易云音乐在近期更新的5.9.1版本中，在“我的”模块的金刚区中整合了其产品下几乎
所有的音乐推荐入口，包括：以前侧边抽屉里的驾驶频道、亲子频道、古典专区、小冰电
台；在首页里的私人FM、每日推荐、排行榜、歌单；以及新推出的私藏推荐等。


一、产品特点
网易云音乐是一款主打发现和分享的社交型音乐的APP，通过日推、私人FM以及推荐歌
单等个性化推荐的方式，满足用户的个性化听歌需求。通过加入社交元素，如评论，点赞
，动态等方式，满足用户分享需求。因此相较于其他音乐APP的用户，网易云音乐的用
户整体上对发现音乐以及分享音乐的需求更强烈。
二、需求背景
1. 用户的分享需求
在产品当前的功能架构中（5.9版本），平台帮助用户发现音乐的功能组成，用户使用该
功能的场景以及该功能的特点如下图所示：


可以发现：除推荐歌单外，几乎所有的推荐结果都是由平台直接推荐给用户的，用户在其
中的参与感并不强。并且相比其他音乐APP，网易云音乐的用户有更强的分享音乐的需求
，而目前产品所能提供给用户分享的途径很有限，其主要分享场景如下：
（1）A发现了一首自己非常喜欢的音乐，并且认为自己的朋友B也会喜欢，于是通过私
信或微信等渠道将这首音乐分享给了B。
（2）C发现了一首自己非常喜欢的音乐，于是他把这首歌分享到朋友圈/微博并配上一段
文字来推荐，希望能让更多的朋友也能喜欢上这首音乐并且与自己交流。
（3）D很喜欢音乐，有较好的听歌品味，于是他将一部分音乐整理成了歌单，这份歌单
在平台上被很多用户收藏了，他也借此认识了很多与他有相似品味的小伙伴。


可以发现：这类有分享需求的用户都是希望通过分享音乐，来吸引其他人与自己产生互动
，以获取他人对自己的认同感，满足自己情感方面的需求。比如当用户非常想向他人安利
一首音乐的时候，越多的人与他产生互动，那么他的被认同感就会越强。
然而，就目前平台提供的分享方式来看，用户所能获得的认同感是非常有限的。因为用户
目前无论是分享到平台内的动态中还是其他社交平台，其曝光量都非常有限，并且由于被
分享的用户中与该用户有相似听歌品味的比例也较低，因此产生互动的转化率也会非常低
。
另一方面，网易云音乐相较于其他音乐APP，其喜欢小众音乐的用户群体也更多，而这其
中的大部分用户也希望自己所喜欢的小众音乐能被越来越多的其他人所喜爱（低评论数的
音乐评论区中常可以看到将这首音乐比作宝藏的评论，他们希望这个宝藏能别其他人所发
掘），因此这类用户的分享意愿也更高。
2. 用户发现音乐的需求
无论是明星代言广告、淘宝直播还是抖音的视频同款，用户在这些渠道产生的转化不仅仅
是因为认为这款商品好，还有一部分原因是因为基于对推荐人的信任。因为推荐人对商品
的背书，更容易让被推荐人对该商品产生好感并产生转化。
在音乐上也是一样，当一首音乐在推荐给用户的过程中，融入了一些由其他用户创造的
UGC内容后，会让听众在除了评判这首音乐是否在听觉上符合品味外；还有可能会因为
其UGC内容引起听众情感的共鸣，从而让听众更容易对这首音乐产生好感。
三、需求方案
为用户提供一种新的音乐推荐平台，平台用户分为两种：第一种为有音乐分享意愿的用户
，称为推荐用户；第二种为没有明确听歌目的，且喜欢发现新音乐的用户，称为被推荐用
户。
平台内所有音乐由推荐用户基于网易云音乐现有乐库自主选择，并通过一段录音说明推荐
理由，由后台筛选并配置好GIF后再进入平台的推荐列表。
首页推荐利用配置好的GIF、歌词和评论轮播以及推荐的音乐，给用户在视觉和听觉上创
造一种类短视频的体验。同时通过推荐用户头像，推荐用户录音、红心、点赞以及评论，

让被推荐用户与推荐用户之间能联系起来，让被推荐用户更容易产生基于音乐外的在情感
层面上的共鸣。

五、页面说明
1. 首页-播放和暂停页

前置条件：
暂停状态
从我的或首页打开‘私藏推荐’小程序 播放状态下点击空白区域
播放状态
暂停状态下点击空白区域 上下划动切歌后
页面逻辑：顶部我要推荐按钮：点击我要推荐，跳转至推荐页。
顶部推荐用户信息：
点击头像，跳转至用户个人主页。 点击[+]，直接关注该用户，并且[+]消失。 若被推荐
用户在之前已经关注该推荐用户，则不显示[+]。 当正在播放用户录音时，[录音播放状态
]处为波动的音轨以及该录音时长。 录音播放完后，[录音播放状态]处的信息消失。 点赞
后，点赞icon变红，赞数+1，再次点击取消点赞。
评论轮播
轮播从录音播放完后开始，每条评论持续4秒，中间间隔2秒，效果都为渐变。 暂停时
评论依然轮播。 评论按评论区顺序依次轮播，需要剔除掉掉回复他人的评论。 当评论字
符超过90时，后面的用“……”代替。 当轮播完全部评论时音乐仍没播完，评论再从头
开始轮播。 点击评论轮播可展示评论详情，展示评论所有内容，点击用户信息栏区域都
可跳转至用户个人主页。
歌词轮播
若音乐有歌词，则按照音乐的时间轴轮播歌词；若没有歌词，则留空。 歌词最多展示四
行 若歌词为外文且有翻译，将外文放上面，中文翻译放下面，其中外文最多三行，中文
最多一行，超过字符限制用“……”代替。
红心
红心下的计数表示在该私藏推荐下被红心的次数，当红心超过1w时，显示为1w+。 点
击红心后将该音乐添加到”我喜爱的音乐”歌单中，红心数+1，红心变红，弹出浮层提

示“已添加到我喜欢的音乐”。 如果在听到这首私藏推荐前已红心了这首歌，切换到的
时候红心就是红的。 已经红心的音乐若再点击一次，红心数-1，红心变白，该音乐从“
我喜爱的音乐”歌单中移除，弹出浮层提示“已取消喜欢”。
目前的红心计数逻辑感觉有点问题： 在用户已红心的情况下如果取消红心，只会在前端
页面上给用户看到红心数-1的反馈，但实际上该红心数在后台是不会减少的，所以只要用
户多次红心并取消红心，则实际被红心的次数是会一直上涨的。 如果说这是网易云音乐
PM故意设计的，那我觉得具体可解释的原因只有两个： （1）利用人的从众心理，希望
用更高的红心数吸引用户点红心。 （2）由于红心下的计数表示的是在该私藏推荐下被红
心的次数，并且红心的状态代表用户是否有将这首音乐红心。所以如果用户在听到这首私
藏推荐之前已经红心了这首音乐，这个红心是不应该被计入到下面的红心计数里的。但是
如果用户在此时取消了红心，而下面的数字不发生变化的话，会让用户觉得很奇怪。为了
让用户的体验保持统一，所以在前端的计数上做了一个类似安慰剂按钮的设计，即通过数
字的变化给用户一个反馈，但实际计数在后台是不变的。
评论
点击评论后，从底部弹出评论区浮层。 评论数为该音乐的评论数，若评论数大于999，
则显示为999+。
为什么红心数要大于1w才变成1w+，而评论大于999就变成999+了呢？ 我认为它之
所以叫“私藏推荐”就是因为平台更鼓励的是用户对小众音乐的分享， 而小众与大众的
分界线就在于是否是999+，所以只要显示到999+就可以达成目的了。 而红心数表示的
是在该私藏推荐下被红心的次数，因此这个数字将会直接影响到推荐用户的成就感以及被
推荐用户对这首歌的好坏的第一印象。所以提供一个更高的上限可以给双方用户都带来更
好的情感上的体验。
分享：点击转发图标后，弹出分享浮层，与普通音乐分享的组件一样。
歌手信息：点击歌手及歌曲信息区域，跳转至歌手列表。
切歌逻辑

上划切换至上一首私藏推荐，下划切换至下一首私藏推荐。 切换后为播放状态。 当听完
一首私藏推荐后，自动跳转至下一首私藏推荐。
一首私藏推荐播放完后我目前遇到过这几种情况： 自动切换到下一首（理想的正常情况
，大概只会持续两首的样子） 自动切换到上一首（不知道为什么会这样，有时不息屏的
状态下就会切换到上一首，然后再听一次……） 停止播放（常见情况，一般是自动切过两
首后到第三首播完后就停止播放了 除了第一种之外另外两种都是不正常的，在评论里也
看到有其他用户反馈过这个问题。

2. 评论轮播详情页

前置条件：点击评论轮播图弹出该评论详情浮窗。
页面逻辑
点击用户信息区域，跳转至该评论用户的个人主页。 点击评论详情区域之外的任何区域
都直接返回播放页。
关注
点击关注关注该评论用户，按钮变红。 若用户在之前已关注该用户，则关注按钮默认为
红。 关注按钮变红状态下再次点击则恢复默认状态，并取消对该用户的关注。
点赞
点赞后点赞按钮变红，点赞数+1，若点赞数大于999，显示为999+。 若用户在之前已
对这条评论点赞，则点赞按钮默认为红。 点赞按钮变红状态下再次点击则取消点赞，点
赞数-1，999+不变。
3. 评论详情页

前置条件：点击评论图标，从下弹出评论详情页浮层。
页面逻辑
页面逻辑、内容与排版完全与该首歌的评论一致，只对配色进行了调整。 点击关闭图标
或任意除评论区外的其他区域，评论区向下收起，返回到播放页。
优化建议 用户在该私藏推荐中产生的评论有一部分是针对该推荐人的，比如夸“小哥哥
声音好有磁性之类的”。 （1）如果不是通过该私藏推荐看到这条评论的用户肯定会产生
一些疑惑，所以建议对这类评论增加一个“私藏推荐”标签标识一下，并且其他用户可通
过该标签跳转至该私藏推荐。 （2）推荐用户在完成该私藏推荐后，他只能通过点赞和红
心数了解到这首私藏推荐被其他用户所喜爱的数据情况，但没法与他们互动。建议将在该
私藏推荐下产生的评论置顶到该私藏推荐的评论区中并通知该推荐用户有人评论了你的私
藏推荐，让推荐人可以与被推荐人之间能进一步互动。

4. 我要推荐页

前置条件：播放/暂停页点击左上角[我要推荐]跳转。
页面逻辑
投稿时间限定在周日到周四的16：00-24：00，若在投稿时间点击推荐我的私藏音乐跳
转至选曲页。 若在非投稿时间点击[推荐我的私藏音乐]，弹出浮层提示“私藏推荐将在每
个周日到周四的16：00-24：00开放投稿哦~请到时候再来”，持续时间2秒，效果为
渐变。 若在投稿时间第二次投稿，弹出浮层提示“今日已投稿，请明天再来”，持续时
间2秒，效果为渐变。 点击[我的历史推荐]跳转至历史推荐列表页。 点击返回图标，返
回到暂停页，切换成一首新的推荐。
5. 选曲页-搜索联想页-搜索结果页

前置条件
在规定时间内，从我要推荐页点击[推荐我的私藏音乐]跳转至选曲页。 在选曲页点击输入
框进入搜索联想页。 在键盘上点击[搜索]或在搜索联想浮层中选择后跳转至搜索结果页。
页面逻辑
搜索输入框
点击搜索输入框，弹出键盘用于输入文字，搜索框下弹出搜索联想结果浮层，联想结果跟
着输入结果变化。 在搜索时点击取消图标退出搜索状态。
红心歌单列表
列表内最多包含50首红心的歌曲，通过滑动控制列表。 点击歌曲对应的区域，跳转至这
首歌的录音页。
就目前体验来看，选曲页的我的红心歌单列表有点鸡肋，主要问题有： （1）我的红心歌
单列表里音乐的选取和排序逻辑不明（可能是根据最近红心排序的，而且有很多歌评论数
很多)，感觉推荐效果不好，最终推荐的音乐更多的还是用搜索完成选择。 （2）由于很多
被推荐的音乐都是重名的或英日韩文，所以用户在选曲前还希望能试听一下，看看是不是
自己想推荐的那首歌。但现在只能选曲后先录制试听了才知道是不是自己想要选的那首，
流程比较复杂。 优化建议： （1）优化我的红心歌单列表排序和选取策略，比如加入用户
听歌次数作为选曲和排序依据，剔除掉评论数多的音乐（私藏推荐鼓励分享小众音乐）。 
（2）在选曲列表里加入预览功能，比如点击专辑封面，即可播放这首音乐。
搜索结果列表
点击搜索输入框，弹出键盘用于输入文字，搜索框下弹出搜索联想结果浮层，联想结果跟
着输入结果变化。 点击歌曲对应的区域，跳转至这首歌的录音页。
搜索结果列表优化建议：
目前用户搜索时搜索结果展示太多，让用户难以找到自己想要推荐的那首歌。因此可以在
搜索结果中优先展示之前被红心过或听过次数多的音乐。

其他：点击返回图标返回至我要推荐页。

6. 历史推荐页

前置条件：从我要推荐页点击[我的历史推荐]跳转。
页面逻辑
历史推荐音乐按时间顺序排序 推荐时间格式为mm/dd/yyyy 点击返回跳转至我要推荐页
推荐状态说明
当推荐状态为已投稿时，左侧状态灯为灰色，右侧为点赞数，播放数以及红心数。 当推
荐状态为已退回时，左侧状态灯位红色，右侧为退回原因。 当推荐状态为精选推荐时，
左侧状态灯为绿色，右侧为点赞数，播放数以及红心数。 当推荐状态为已投稿或已退回
时，点击歌曲所在框内区域，跳转至播放页。该播放页内无论点击除返回图标以外的任何
地方，都只能控制播放/暂停状态，UI上将播放页变暗，在界面正中央出现文字提示，样
式如下图：
当推荐状态为精选推荐时，点击歌曲所在框内区域，跳转至播放页，该播放页交互以及界
面与正常播放页一致，只是不能上下划动切歌。
计数逻辑说明


点赞数为用户在播放页/暂停页为该私藏推荐的点赞数。 播放数为用户在该私藏推荐播放
状态下出现的次数，若暂停状态下就切换了不计数，在没有切换到其他私藏推荐的条件下
，至多只计一次播放量（多次暂停并播放只算一次）。

7. 准备录制页

前置条件：在选曲页已经通过搜索或我的红心歌单列表选择了要推荐的音乐。
页面逻辑
点击录制后，弹出录音权限申请提示，允许后跳转至正在录制状态。 点击返回图标返回
到选曲页。

8. 正在录制页

前置条件：从准备录制页点击录制后跳转。
页面逻辑
跳转过来后直接开始录制，录制时长最多20秒，以倒计时形式提醒剩余时长。 点击停止
或录制时间到20秒后跳转至录制完成页。 录制完成后自动将录音与原音乐合成，有录音
的部分音乐音量调低。 点击返回图标返回到选曲页（不是返回到准备录制页）。

9. 录制完成页

前置条件：正在录制页中点击停止或录制时间到20秒后跳转。
页面逻辑
点击播放键，开始播放合成后的音乐。 若音乐在后台还未合成完成就点击播放键，弹出
相关提示toast，持续2秒。 点击投稿，弹出确认是否投稿的浮层。 确认投稿后，显示
上传进度；若上传成功，弹出上传成功toast，持续2秒，跳转至历史推荐页。 确认投稿
后，显示上传进度；若上传失败，弹出上传失败toast，持续2秒。 点击重新录制，跳转
至正在录制页，直接开始录制。 点击返回图标，返回至选曲页。

10. 分享页

前置条件：在播放/暂停页点击分享图标后弹出该分享组件。
逻辑说明
分享与平台内原有的分享组件保持一致，只修改分享内容。 无论是分享到平台内或是平
台外分享，打开分享连接后都是用h5打开该私藏推荐。
11. 消息通知
逻辑说明：私藏推荐审核
若审核通过，由云音乐私藏君私信用户告知推荐已通过。 若审核未通过，由云音乐私藏
君私信用户告知推荐未通过，并给出原因。
通知说明
当被推荐用户点红心或点赞后，该私藏推荐的推荐用户在通知中收到相应通知消息。 若
被推荐用户点了多次红心或赞，推荐用户也会收到多个通知。
六、总结
之所以选择私藏推荐作为给自己布置的第一篇PRD，一方面是觉得如果从网易云音乐整
体去倒推PRD难度太大，所能关注到的逻辑也很有限，所以选择了其中的一个小功能，
希望能关注得比较全面（当然后台的一些逻辑我是没法知道的……）。另一方面是因为觉
得这个功能非常有新意，所以想借着倒推PRD的机会，去猜测当时做私藏推荐的PM是
怎么想到要做这样一个功能，并且要做成现在这种形式。
作为一名产品小白，我的很多理解和推论都还不够成熟，因此也希望借助人人这个平台各
位大佬们的意见和建议，帮助我早日称为一名合格且优秀的产品经理！
最后借着这个机会，还想跟大家一起讨论一下为什么私藏推荐的播放页要做成类短视频的
样式，而不是普通播放器的样式？

我自己的看法是：做成类短视频的播放页可能是想增加用户停留在播放页的时间，让用户
的注意力能更集中一些，从而提高用户红心、点赞、评论、关注等行为的转化率，所以评
论和歌词的轮播我认为也是提高用户注意力的一种方式。
如果我是PM，可能我能想到让用户以语音的形式推荐，但其展示形式应该就还是一个播
放器了。
