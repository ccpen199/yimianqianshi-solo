第⼀一步:是否⽴立项1. ⽼老老⼤大让⽴立项不不代表真的值得⽴立项 ——经验、直觉不不代表真理理 2. 问题---⽬目标—论证—资源(性价⽐比)---研发周期---测量量⽅方法不不能发现问题的PM是不不合格的建⽴立指标模型,如果需求不不紧扣这些模型,那估计哪错了了 不不能量量化的⽬目标不不值得做没有推导过程订⽴立的⽬目标⼩小⼼心⽆无法完成不不能评估⼯工作量量的需求没法进⾏行行性价⽐比评估3. 这部分只所以重要原因如下如果模型评估,发现⽬目标太低,性价⽐比太低,你就不不⽤用写⽂文档了了 如果发现⽬目标可以特别⾼高,恭喜你资源向你 奔来 如果⼀一开始没有想明⽩白就开始动⼿手写⽂文档,50%概率你会写偏 

第⼆二步:搭建框架 将产品所有功能进⾏行行合理理分解和排序,确定PRD各节标题。基本规则: 按⻚页⾯面元素分解：上—>下、左—>右按⽤用户操作步骤分解 提交—>展示—>展示后编辑按在系统中所处位置分解前台⻚页⾯面—>⽤用户管理理后台—>官⽅方管理理后台按功能主次分解主要功能—>次要/附属功能(特殊权限、⼴广告位、 其它相关说明等) 

第三步：梳理理主线按照已确定的PRD章节顺序,⽤用关键示例例图+简要⽂文字描述  的⽅方式对主要功能点进⾏行行说明。此步骤只关注功能主线,不不⽤用有过于详细的描述,也不不⽤用涉及 各种特殊状态和细节的处理理。使产品所包含的主要功能在PRD中有完整体现。 在主要功能点的整理理过程中,对PRD的结构及时进⾏行行合理理调整 时刻回顾下主线跟⽬目标关系此步骤完成之后,可与开发⼈人员进⾏行行初步沟通 

第四步:填充细节对产品功能及其它相关需求进⾏行行完整说明包括所有操作流程、判断逻辑、权限区别、⻚页⾯面效果、特殊 状态处理理、错误提示、已 有功能说明等 ——此步骤完成之后可发起内部需求评审细节说明通常会占到PRD篇幅的70%以上。⼀一份细节清晰完整 的PRD是项⽬目顺利利进⾏行行 的有⼒力力保障,也是PM对产品理理解和 掌控程度的重要体现

1.任何⻚页⾯面都要说明“从哪来,到哪去” ⻚页⾯面⼊入⼝口 PC时代有title、DESC、KEYWORD ⻚页⾯面初始状态 ⻚页⾯面展现和功能细节,按⼀一定顺序描述 各链接点击效果、指向地址、打开⽅方式、刷新⽅方式 浮动层具体策略略是否⾃自动关闭右上⻆角是否展示关闭按钮,点击效果如何 若在浮动层中可打开新⻚页⾯面,原浮动层是否关闭 关闭后是否刷新⻚页⾯面...... tips:如果能⽤用低保真交互原型进⾏行行表现,可以丰富你的⽂文档。

2.不不要只考虑普通⽤用户 若⻚页⾯面对不不同权限⽤用户有不不同展示和功能,要完整 说明并提供准确示意图。eg：酒店o2o系统saas系统

3.形成条件反射的错误提示输⼊入为空 包括输⼊入空格/空字符串串超过字数上限 前台以汉字数提示,技术上以字符数限制含特殊字符 可⽤用字符集⼀一般分常⽤用字符(汉字、字⺟母、数字、下划线)和GBK字符两种，由输⼊入内容的应⽤用范围来定含过滤词 需明确过滤词表其它输⼊入⽆无效的情况 有特殊格式要求/不不能重复/有特定范围限制/ ⽆无提交权限 退出登录/被封禁/不不符合权限要求 ...... 

4.输⼊入框⾥里里陷阱多是否可以为空是否有初始内容,是否默认选中⼤大⼩小写/全半⻆角/繁简体是否转换 任何输⼊入框都需要字数上限 允许的字符集空格出现在⾸首尾和中间部分,或者连续多个空格的各⾃自处理理⽅方式多⾏行行⽂文本框的连续空⾏行行、不不连续空⾏行行、空格、tab键、回⻋车 键等处理理⽅方式是否允许快捷键控制
