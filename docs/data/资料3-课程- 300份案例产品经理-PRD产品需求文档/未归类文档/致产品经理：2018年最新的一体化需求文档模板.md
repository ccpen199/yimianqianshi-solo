# 致产品经理：2018年最新的一体化需求文档模板

本文作者结合自身经验，输出了一份最新的一体化需求文档模板，希望对你有用~enjoy~
概要页
一般会写出文档为谁而写，能够提供什么，以及本次项目的背景，参考了哪些资料写出的
。


功能页
具体到每一项功能，完整的介绍功能为谁设计，解决他的什么问题，目的是让运营、测试
、开发同事可以选择自己关注的方面了解下去。
作为一名文笔不好的产品经理，没办法给大家带来优雅的描述，所以经常被人投诉文档之
中前言不搭后语，辛辛苦苦写了几十万字的文档愿意读完的人寥寥无几，在2017初在人
人都是产品经理上看到了一体化需求文档，于是就在这一整年的工作中不断的融合这份文
档，现在回馈给大家，当然这里面还有一些不到位的地方，比如控件使用不合理，连线不
美观等也希望大家能教教我写的更好。
rp文件在最后，不足之处还请大家多多指点。
系统概述
这是关于产品的全局描述，一般会写出这个产品的背景，解决谁的问题，这些问题属于谁
，而又在哪些场合发生，这样写的目的就是让开发、测试等不接触市场的同事能够理解自
己所做的项目，促进开发过程的连续性，会分为概述-产品定义-目标用户-应用场景-名词
解释-产品路线-会议存档-团队分工-功能需求-非功能需求：
概述：写出编写文档的目的； 产品定义：具体写出产品解决问题的方式； 目标用户：介
绍购买本产品或操作本产品的人物特征； 应用场景：描写目标用户在不使用和使用产品


时解决问题的方式； 名词解释：介绍系统里定义的一些名词，操作等； 产品路线图：描
述产品的过去，现在，将来版本的信息； 会议存档：一般直接把白板的照片贴进来，或
者有秘书整理的会议纪要； 团队分工：介绍运营-产品-项目-开发-测试-总负责人信息； 
功能需求：点击后直接跳转对应的功能页面； 非功能需求：写出安全性，访问，设备性
能等。


以上是对整套系统的全局说明，在整个2017年产品经理的工作就是逐个填写10个菜单
，与各个部门协作完成一个系统的产品设计工作，接下来是一个功能的设计工作流程。
功能
1. 概述-具体到解决哪一个问题；
2. 使用角色-写出谁来操作这个功能；
3. 前置条件-在使用这个功能前，系统或者用户需要做什么；
4. 后置条件-影响到了哪些模块；


5. 业务流程-可以使用Visio画出业务流程；
6. 页面流程-原型界面+数据+操作+状态流程；
7. 高保真-放置高保真图片；
8. 交互规范-一般写些用户体验优化的规则；


在确定一个功能的时候会按照上面8个菜单顺序进行完善；
通过对系统和功能的完善就是整个2017年的产品工作，在配上一份产品工作的脑图；
