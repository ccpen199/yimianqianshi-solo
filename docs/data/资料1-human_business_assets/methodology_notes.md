# 项目业务卡片总结方法说明

更新时间：2026-04-25

## 目标

这个资料库的目标不是写创业方案，也不是只列项目名称，而是把大量人类业务沉淀成可用于代码任务、PRD、功能迭代、Bug 修复、测试设计和业务理解的“项目业务卡片”。

每张卡片都尽量回答五件事：

1. 这个项目解决什么业务问题。
2. 业务里有哪些核心角色。
3. 业务流程如何流转。
4. 系统需要沉淀哪些功能资产。
5. 甲方可能提出什么代码任务。

## 总结思路

我使用的是“行业分类 + 业务流程 + 软件系统形态 + 甲方任务类型”的交叉归纳法。

第一层是行业覆盖：
- 农业、制造、物流、零售、电商、金融、保险、医疗、教育、地产、物业、餐饮、文旅、交通、能源、环保、政务、公益、内容、娱乐、企业服务、开发者工具、AI 工具等。

第二层是业务对象：
- 人、货、钱、设备、订单、合同、工单、内容、数据、证照、资产、项目、任务、风险、权益、积分、库存、账单、审批、报告。

第三层是系统流程：
- 建档、申请、审核、派单、执行、验收、结算、提醒、预警、报表、归档、复盘。

第四层是代码任务映射：
- Bug 修复、0-1 代码生成、Feature 迭代、代码理解、代码重构、工程化、代码测试、数据可视化、后端 API、Web 前端、全栈应用、移动端、桌面 GUI、命令行工具、AI/ML、科学计算、自动化脚本。

## 依据来源

这些卡片不是逐条实时联网复制某个网页，也不是只靠随机发挥。主要依据来自三类资料和经验结构：

1. 标准行业分类：
   - NAICS 行业分类
   - GICS 行业分类
   - BLS Industries at a Glance
   - 常见国民经济行业分类思路

2. 公开产品与业务形态：
   - SaaS、CRM、ERP、MES、WMS、TMS、LMS、CMS、BI、工单、支付、会员、订阅、交易平台、撮合平台、预约平台、风控平台、数据平台、AI Agent 平台等通用软件形态。
   - 常见产品来源包括 Product Hunt、G2、Capterra、Futurepedia、Zapier、n8n、Make、Gumroad、Patreon、Substack 等公开产品生态中常见模式。

3. 行业通用业务流程：
   - 例如零售的“商品-库存-订单-支付-售后”。
   - 制造的“BOM-排产-工序-质检-入库”。
   - 金融的“KYC-风控-审批-放款-还款-贷后”。
   - 医疗的“建档-预约-诊疗-处方-报告-随访”。
   - 物流的“下单-调度-运输-签收-异常-结算”。
   - 政务的“申请-材料-审批-补正-办结-归档”。

## 是否每次实时发挥

不是纯实时发挥。

实际做法是：

- 先用行业地图确定覆盖范围，避免只围绕 AI/SaaS。
- 再按各行业的典型业务流程抽取系统项目。
- 然后把每个项目转换为统一卡片结构。
- 最后把它映射到甲方常见代码任务类型。

其中项目名称和部分组合会根据已有业务模式进行重新组合，例如：

- “物业报修缴费系统”来自物业行业常规流程。
- “SaaS 套餐计费系统”来自 SaaS 商业化通用模块。
- “AI Agent 任务队列系统”来自 AI 工具和工作流平台常见架构。
- “供应商 ESG 评估系统”来自采购、合规和 ESG 管理的交叉业务。

所以它是“结构化归纳 + 行业经验迁移 + 软件功能抽象”，不是照搬单一来源。

## 使用的工具

本轮沉淀主要使用：

- 本地文件写入：`apply_patch`
- 文件检查：`ls`、`wc -l`、`rg`
- 资料保存目录：`data/human_business_assets/`
- 输出文件格式：Markdown

当前文件是人工结构化整理结果，没有写自动生成脚本，也没有把它做成数据库。后续如果要检索、筛选、去重、按行业生成 prompt，可以再转成 JSON/CSV。

## 为什么每张卡片都用同一结构

因为你的目标是拿它去完成截图里的任务类型。如果每个项目只写一句话，无法支持：

- 0-1 代码生成
- Feature 迭代
- Bug 修复
- 代码理解
- 代码测试
- 后端 API
- Web 前端
- 全栈应用
- AI/ML 应用
- 数据可视化

统一结构可以直接变成业务 prompt、PRD、测试用例或代码任务。
