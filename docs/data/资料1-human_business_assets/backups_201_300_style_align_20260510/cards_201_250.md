# 项目业务复杂代码生成需求 201-250

## 201 综合电商 Web 全栈平台

请从综合电商 Web 全栈平台的真实交易闭环出发，先把消费者、商家、仓库、客服、运营和财务的协作链路梳理清楚，再把首页、详情、购物车、订单、支付、物流和售后落到同一套后端事实里。
Web 零售系统要服务多角色协同，核心对象包括商品目录、SKU、价格策略、购物车、优惠券、订单、支付流水、物流包裹、售后单和用户复购画像。复杂度应落在促销叠加、库存预占释放、支付回调幂等、拆单发货、逆向售后和报表穿透上，前端不能自行拼价格、库存或售后状态。
1. 运营在后台维护类目、商品、SKU、会员价、限购、优惠券和活动档期，后端校验价格叠加关系和库存来源后发布到 Web 端。
2. 消费者搜索商品、进入详情并选择规格时，后端按地域、会员等级、SKU库存、活动时段和限购规则返回可买组合，前端只展示后端允许的购买入口。
3. 消费者加入购物车后，后端合并同 SKU、重算促销、标记失效商品和凑单建议，购物车金额、优惠分摊和可下单状态都写入可追溯记录。
4. 消费者提交订单时，后端锁定库存、生成订单主表和明细表，计算运费、优惠、积分和拆单方案，并把待支付倒计时同步到用户端。
5. 支付渠道回调后，后端按订单号、金额、渠道流水和当前状态做幂等处理，成功则创建仓库履约任务，超时或差错则释放库存并生成支付异常记录。
6. 仓库拣货、复核、打包和发货都要回传包裹节点，用户端、客服端和商家端看到同一份物流进度，客服只能基于履约节点发起改址、催发或拦截。
7. 用户按订单明细发起退款、退货或换货时，后端校验签收状态、责任原因、逆向物流和可退金额，售后结果同步影响商家结算、库存回补和用户权益。
8. 运营报表按商品、渠道、优惠、履约和售后维度统计，任一指标都能下钻到订单、支付、包裹、售后和操作日志。
9. 异常必须覆盖超卖、优惠冲突、支付超时、地址不可达、拆单失败、退款不到账和重复提交，每类异常都要有阻断、补偿、人工处理和审计路径。
落地要求：统一实现为浏览器 Web 全栈业务系统，不要做独立 App、移动端安装包、小程序壳、桌面客户端或纯前端静态页面；必须包含 Web 前端、后端 API、SQLite 数据库和必要的管理/运营视图，所有核心动作都要落库并可复查。
组件约束：不要生成容器化配置或非必要外部服务，数据库默认优先使用 SQLite 文件数据库（例如 data/app.sqlite）以最快打通完整业务链路；如果业务确实强依赖 MySQL、PostgreSQL、Redis、消息队列或对象存储等特定组件，可以引入，但必须说明依赖原因、配置入口、启动方式和本地降级方案。
端口约束：前后端端口必须优先围绕项目目录名称中的数字本身规划，但端口号必须是 0 到 65535 之间的合法数字。四位项目号例如 may-8744，端口必须包含 8744，可使用 87441、87442 或 18744、28744 这类合法前后缀组合；五位项目号例如 may-11132，不能拼成 111321 这类超过 65535 的非法端口，必须取后四位 1132 作为端口前后缀，可使用 11132、21132、11321、11322 等合法端口；若目录数字较短，可使用该数字补位后加 01/02（如 may-71 使用 7101/7102）。统一写入 .env 或配置文件，并同步 CORS、代理和回调地址，避开 3000、5173、8000、8080 等常见端口。
验收约束：完成代码后必须实际启动后端和前端，打印访问地址，并通过关键页面操作和接口请求验证主业务链路已调通。
前后端启动方式统一使用后台启动，关闭终端不影响程序运行，只有手动关闭对应端口进程才算完成关闭。
不要提问我问题，遇到脚本问题、执行问题、权限问题等都允许你自行解决。
在执行阶段，涉及项目内删除操作时无需再征求我的同意。
按照需求功能的顺序，每一轮都优先考虑搭建核心功能，在逐步的考虑其他功能

## 202 生鲜到家 Web 全栈平台

请把生鲜到家 Web 全栈平台 写成前置仓即时履约系统，不要把它做成普通商城换皮。核心要围绕地址服务范围、批次库存、称重差异、拣货缺货、骑手时效和冷链异常展开。
系统服务用户、前置仓拣货员、配送员、采购、客服和运营，核心对象包括生鲜 SKU、产地批次、效期库存、配送时段、拣货任务、称重记录、缺货补偿、配送轨迹和售后退款。规则重点是时段容量锁定、临期先出、称重计价、缺货替换、冷链温控和损耗复盘。
1. 用户选择地址后，后端根据服务范围、前置仓覆盖、配送容量和实时库存返回可买商品、最快送达时段、运费和起送门槛。
2. 用户下单时，后端同时锁定商品批次和配送时段容量，订单明细记录预估重量、计价方式、可替换偏好和缺货补偿规则。
3. 拣货员按货位和效期领取任务，扫码确认批次并称重复核，系统根据实称重量自动计算补差、退款或待用户确认金额。
4. 发生临时缺货时，后端按用户偏好生成替换、部分退款、补货等待或整单取消方案，客服端能看到缺货原因、影响商品和补偿金额。
5. 骑手接单后，后端记录出仓、到达小区、联系用户、签收和异常节点，超时、温度异常或轨迹偏离会进入配送预警。
6. 用户签收后可按商品发起腐损、少件、重量差异或超时售后，系统按责任原因决定退款、补发、优惠补偿或人工复核。
7. 采购和运营端根据损耗率、缺货率、时段满载率、客诉原因和批次周转分析补货计划，报表可追到仓、批次、订单和骑手。
8. 异常必须覆盖称重差价、临时缺货、骑手超时、冷链温度异常、用户拒收、退款争议和前置仓爆单。

## 203 药品配送 Web 全栈平台

药品配送 Web 全栈平台 必须把处方合规、药师审方、批号库存、冷藏履约和实名签收串成受监管链路，前端不能绕过审方和限购规则直接完成交易。
系统面向购药用户、药师、药店仓库、配送员、客服和监管查询人员，核心对象包括药品目录、用药人档案、处方凭证、审方记录、库存批号、冷藏配送单、实名签收和合规审计日志。后端要承担处方有效期、禁忌剂量、限购、批号召回、冷链时效和隐私授权。
1. 用户选择药品时，前端按后端返回的处方药、OTC、冷藏药、限购和用药人条件展示购买限制。
2. 用户上传处方或在线问诊凭证后，后端生成待审方任务，药师端展示药品、剂量、诊断、用药人年龄和历史购药记录。
3. 药师审核通过后，系统锁定批号、效期和库存；审核驳回时记录禁忌、剂量、处方不清或资料缺失原因，并推送给用户补正。
4. 用户支付成功后，后端核验渠道流水和订单状态，药店仓库按批号拣货，冷藏药品自动附带保温箱、温度范围和最晚送达时间。
5. 仓库打包扫码时，系统校验批号是否召回、是否近效期、是否需要冷藏，命中风险则阻断出库并要求换批号。
6. 配送员交付时完成实名核验、签收拍照或拒收记录，签收结果同步给用户、药师、客服和监管查询端。
7. 客服处理退款、补送或用药投诉时，只能基于审方记录、批号、配送温度和签收状态发起，不能直接改写合规链路。
8. 异常必须覆盖处方过期、管制药限制、药师驳回、批号召回、冷藏配送失败、实名不符和隐私越权。

## 204 本地生活团购平台

本地生活团购平台要围绕商家套餐、用户购券、门店核销、退款评价和财务结算展开，不要只做商品列表。
系统服务消费者、商家、门店核销员、平台运营、客服和财务，核心对象包括商家门店、团购套餐、适用规则、券码、核销记录、退款单、评价和结算账单。后端规则要覆盖券码唯一性、防重复核销、有效期退款、门店分账和刷单识别。
1. 商家在后台创建套餐，录入适用门店、可用日期、库存、预约要求、不可退条件和退款规则，平台运营审核后才允许上架。
2. 用户购买套餐后，后端生成唯一券码、支付流水和有效期，前端展示预约入口、门店范围、不可用日期和退款边界。
3. 用户到店核销时，核销员扫码提交门店、设备和券码信息，后端校验状态、次数、有效期、适用门店和重复核销风险。
4. 核销成功后，系统写入核销记录并增加商家待结算金额，同时同步用户评价入口和商家经营看板。
5. 用户申请退款时，后端按是否预约、是否核销、是否过期和活动规则计算可退金额，客服端保留人工例外审批。
6. 财务按核销记录、退款记录、平台佣金和商家账期生成结算单，商家确认后进入打款和对账流程。
7. 运营报表追踪曝光、购买、预约、核销、退款、评价和复购漏斗，所有数字都能下钻到券码和订单。
8. 异常必须覆盖券码转赠限制、过期退款、重复核销、门店停业、套餐变更、结算差异和刷单。

## 205 预约到店服务平台

预约到店服务平台要把服务项目、员工排班、用户预约、订金支付、到店核销、服务完成和改期退款写成可执行闭环。
系统面向消费者、门店管理员、服务人员、客服和运营，核心对象包括服务项目、门店、员工技能、班次、可约时段、预约单、订金流水、核销码、取消规则和评价记录。复杂度落在容量计算、排班冲突、爽约惩罚、改期补偿和核销校验。
1. 门店维护服务项目、服务时长、价格、设备需求和员工技能，后端据此计算每个门店每个时段真实可约容量。
2. 员工排班后，系统锁定休假、培训、已预约和设备占用时段，前端只展示可预约组合，避免用户选到不可服务时间。
3. 用户选择服务、门店、员工或时段并支付订金，后端生成预约单、核销码、提醒计划和取消改期边界。
4. 门店可确认预约、调整服务人员或要求用户补充资料，所有变更都同步给用户并记录原因和操作人。
