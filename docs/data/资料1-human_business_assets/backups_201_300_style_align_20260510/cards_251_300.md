# 项目业务复杂代码生成需求 251-300

## 251 银行开户与客户尽调系统

请按银行开户与客户尽调系统的真实业务现场设计首版全栈实现，先确定谁发起、谁处理、哪些数据会改变、哪些状态必须被后端约束。
系统主要服务业务经办人、复核人员、合规人员、客户、财务和管理者。核心数据对象包括客户档案、申请单、交易流水、风险记录、回执、对账单、审计日志；关键规则要落到适当性校验、名单筛查、资金核算、复核审批、差错处理，保证列表、详情、表单、看板、导出和审计记录读取同一套后端事实。
1. 客户在前端完成提交开户申请和证件，后端校验必填项、唯一性、权限范围和生效条件，生成初始记录并把可处理状态返回到对应工作台。
2. 客户推进经理补充企业与受益人信息，前端提交业务数据，后端按银行开户与客户尽调系统的业务规则更新状态、明细、通知和审计记录。
3. 系统完成身份核验和黑名单筛查后，后端生成风险事件、影响对象、处置等级和升级路径，前端显示阻断原因并提供人工处理入口。
4. 业务人员处理合规人员复核高风险命中时，后端保存审核意见、依据材料、前后状态和下一节点；通过、退回、补正都要反映到提交方页面。
5. 运营人员完成账户运营开立账户并归档材料后，后端冻结关键结果、归档附件和历史版本，同步生成报表口径所需的统计字段。
6. 业务人员推进后续定期尽调触发更新，前端提交业务数据，后端按银行开户与客户尽调系统的业务规则更新状态、明细、通知和审计记录。
7. 异常处理必须覆盖证件过期、受益人不明、名单命中、材料缺失、重复客户、尽调过期，每类异常都要有阻断、补偿、人工复核或回滚路径，并记录影响对象、处理人和最终结果。
8. 验收时按银行开户与客户尽调系统的业务顺序跑通主流程，同时验证异常流程、权限越界、重复提交、历史追溯和报表穿透。
落地要求：统一实现为浏览器 Web 全栈业务系统，不要做独立 App、移动端安装包、小程序壳、桌面客户端或纯前端静态页面；必须包含 Web 前端、后端 API、SQLite 数据库和必要的管理/运营视图，所有核心动作都要落库并可复查。
组件约束：不要生成容器化配置或非必要外部服务，数据库默认优先使用 SQLite 文件数据库（例如 data/app.sqlite）以最快打通完整业务链路；如果业务确实强依赖 MySQL、PostgreSQL、Redis、消息队列或对象存储等特定组件，可以引入，但必须说明依赖原因、配置入口、启动方式和本地降级方案。
端口约束：前后端端口必须优先围绕项目目录名称中的数字本身规划，但端口号必须是 0 到 65535 之间的合法数字。四位项目号例如 may-8744，端口必须包含 8744，可使用 87441、87442 或 18744、28744 这类合法前后缀组合；五位项目号例如 may-11132，不能拼成 111321 这类超过 65535 的非法端口，必须取后四位 1132 作为端口前后缀，可使用 11132、21132、11321、11322 等合法端口；若目录数字较短，可使用该数字补位后加 01/02（如 may-71 使用 7101/7102）。统一写入 .env 或配置文件，并同步 CORS、代理和回调地址，避开 3000、5173、8000、8080 等常见端口。
验收约束：完成代码后必须实际启动后端和前端，打印访问地址，并通过关键页面操作和接口请求验证主业务链路已调通。

系统主要服务学生或老人、家长或家属、老师、管理员、财务和监管角色。核心数据对象包括人员档案、班级床位或线路、服务计划、考勤记录、通知回执、费用账单；关键规则要落到授权校验、容量约束、日常记录、异常升级、费用计算，保证列表、详情、表单、看板、导出和审计记录读取同一套后端事实。
1. 业务人员推进辅导员导入学生名单，前端提交业务数据，后端按智慧校园宿舍管理系统的业务规则更新状态、明细、通知和审计记录。

6. 业务人员推进水电账单按房间分摊，前端提交业务数据，后端按智慧校园宿舍管理系统的业务规则更新状态、明细、通知和审计记录。
7. 业务人员发起退宿时关闭权限和结算后，后端写入资金流水、业务单据和渠道回执，按规则更新可退金额、结算状态和财务对账明细。
8. 异常处理必须覆盖调宿、晚归、门禁异常、维修超时、费用争议、毕业批量退宿，每类异常都要有阻断、补偿、人工复核或回滚路径，并记录影响对象、处理人和最终结果。

系统主要服务设备端、操作员、维修人员、调度、客户和管理者。核心数据对象包括设备档案、传感器数据、任务单、批次、报警、维修记录、执行日志；关键规则要落到设备在线、指令回执、批次链路、故障升级、远程诊断，保证列表、详情、表单、看板、导出和审计记录读取同一套后端事实。
1. 业务人员处理原料入库生成批次，后端同步校验批次、库存、位置、效期和履约状态，成功后更新明细账，失败则生成差异单或补货任务。
2. 业务人员处理生产工单消耗原料批次，后端同步校验批次、库存、位置、效期和履约状态，成功后更新明细账，失败则生成差异单或补货任务。
3. 业务人员推进各工序记录检验结果，前端提交业务数据，后端按质量追溯批次管理系统的业务规则更新状态、明细、通知和审计记录。
4. 业务人员在前端完成成品入库绑定生产批次，后端校验必填项、唯一性、权限范围和生效条件，生成初始记录并把可处理状态返回到对应工作台。
5. 客户处理发货记录流向客户，后端同步校验批次、库存、位置、效期和履约状态，成功后更新明细账，失败则生成差异单或补货任务。
6. 业务人员处理质量问题冻结相关库存，后端同步校验批次、库存、位置、效期和履约状态，成功后更新明细账，失败则生成差异单或补货任务。
7. 客户处理召回按批次追踪客户，后端同步校验批次、库存、位置、效期和履约状态，成功后更新明细账，失败则生成差异单或补货任务。
8. 异常处理必须覆盖批次混用、检验不合格、库存错绑、发货追踪缺失、召回范围扩大、记录补录，每类异常都要有阻断、补偿、人工复核或回滚路径，并记录影响对象、处理人和最终结果。
9. 验收时按质量追溯批次管理系统的业务顺序跑通主流程，同时验证异常流程、权限越界、重复提交、历史追溯和报表穿透。
落地要求：统一实现为浏览器 Web 全栈业务系统，不要做独立 App、移动端安装包、小程序壳、桌面客户端或纯前端静态页面；必须包含 Web 前端、后端 API、SQLite 数据库和必要的管理/运营视图，所有核心动作都要落库并可复查。
组件约束：不要生成容器化配置或非必要外部服务，数据库默认优先使用 SQLite 文件数据库（例如 data/app.sqlite）以最快打通完整业务链路；如果业务确实强依赖 MySQL、PostgreSQL、Redis、消息队列或对象存储等特定组件，可以引入，但必须说明依赖原因、配置入口、启动方式和本地降级方案。
端口约束：前后端端口必须优先围绕项目目录名称中的数字本身规划，但端口号必须是 0 到 65535 之间的合法数字。四位项目号例如 may-8744，端口必须包含 8744，可使用 87441、87442 或 18744、28744 这类合法前后缀组合；五位项目号例如 may-11132，不能拼成 111321 这类超过 65535 的非法端口，必须取后四位 1132 作为端口前后缀，可使用 11132、21132、11321、11322 等合法端口；若目录数字较短，可使用该数字补位后加 01/02（如 may-71 使用 7101/7102）。统一写入 .env 或配置文件，并同步 CORS、代理和回调地址，避开 3000、5173、8000、8080 等常见端口。
验收约束：完成代码后必须实际启动后端和前端，打印访问地址，并通过关键页面操作和接口请求验证主业务链路已调通。
前后端启动方式统一使用后台启动，关闭终端不影响程序运行，只有手动关闭对应端口进程才算完成关闭。
不要提问我问题，遇到脚本问题、执行问题、权限问题等都允许你自行解决。
在执行阶段，涉及项目内删除操作时无需再征求我的同意。
按照需求功能的顺序，每一轮都优先考虑搭建核心功能，在逐步的考虑其他功能

## 300 设备远程诊断与维修派单系统

设备远程诊断与维修派单系统的首版实现要服务真实岗位协作，前端负责录入和展示，后端负责规则判断、状态推进、留痕和统计。
系统主要服务客户、门店、技师、配件仓、客服和财务。核心数据对象包括车辆或设备档案、预约单、检测单、报价单、配件库存、施工记录、质保单；关键规则要落到派工匹配、报价确认、配件扣减、验收支付、质保复修，保证列表、详情、表单、看板、导出和审计记录读取同一套后端事实。
1. 设备端推进设备持续上报运行数据，前端提交业务数据，后端按设备远程诊断与维修派单系统的业务规则更新状态、明细、通知和审计记录。
2. 系统识别故障码和异常趋势后，后端生成风险事件、影响对象、处置等级和升级路径，前端显示阻断原因并提供人工处理入口。
3. 工程师进入远程查看日志并给出诊断页面时，后端按角色权限返回筛选结果、聚合指标和可穿透的明细，前端不得用本地假数据补齐业务结果。
4. 工程师触发需要现场时自动派单匹配工程师后，后端结合容量、时段、技能、距离、优先级和历史记录生成候选方案，确认后锁定资源并通知执行端。

6. 业务人员执行维修完成回传结果，系统校验身份、时间、地点、设备或重量数据，落库操作明细并即时刷新相关角色的进度和异常提示。
7. 业务人员完成故障经验沉淀到知识库后，后端冻结关键结果、归档附件和历史版本，同步生成报表口径所需的统计字段。
8. 异常处理必须覆盖设备离线、误诊断、备件缺货、工程师超时、客户拒修、重复故障，每类异常都要有阻断、补偿、人工复核或回滚路径，并记录影响对象、处理人和最终结果。
