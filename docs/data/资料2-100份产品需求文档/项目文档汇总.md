# 100份产品需求文档项目汇总（产品方案梳理版）
## 一、需求分析文档（19）
### 01. bbs论坛需求分析.md
请从一个真实论坛的日常运营出发，把游客浏览、会员发帖、回帖互动、版块沉淀和版主管理这些动作串成完整闭环，帖子状态、用户权限、版块统计和后台处理都要能互相对上。
1. 用户进入论坛后，首页展示论坛分区、版块列表、最新主题和热门内容，普通访问者可以浏览公开版块和主题详情，注册用户登录后获得发帖、回复、编辑个人信息等操作能力。
2. 版块是论坛内容组织的核心，系统需要支持一级版块、子版块、主题列表、主题详情、回复列表和浏览次数统计。用户发布主题时选择所属版块、填写标题和正文，后端生成主题记录并同步更新版块主题数、回复数和最后更新时间。
3. 回复流程围绕主题展开，用户在主题详情页提交回复后，系统写入回复记录，更新主题最新回复时间和回复数量，并在列表页体现主题热度变化。管理员和版主需要能够删除违规主题、修改内容状态、维护版块结构和处理用户异常行为。
4. 开发实现上要把用户、角色、版块、主题、回复、操作日志拆成稳定数据对象，前端所有按钮和入口都受后端权限约束，避免普通用户访问版主管理能力或后台管理能力。
### 02. BBS论坛需求分析文档.md
这一版BBS论坛更适合按JavaWeb课程项目落地，先把文章、分类、用户、角色、权限和访问统计拆清楚，再让前台浏览发帖与后台维护共用同一套数据和校验规则。
1. 前台以内容访问为主，用户通过大类、小类进入帖子列表，查看帖子标题、正文、发布时间、最后编辑时间、访问次数和锁定状态。发帖时选择分类、填写标题和正文，后端写入文章表并保留创建时间、最后编辑时间和访问计数。
2. 后台以内容和权限维护为主，管理员维护用户、角色、角色权限、大类、小类和文章数据。文章支持新增、删除、修改、关键词搜索、按 ID 查询、查询最近文章、统计总数和访问次数更新。
3. 数据访问采用DAO、DAO代理和DAO工厂组织，文章、分类、用户、角色等对象通过统一的数据访问接口操作数据库。工程上要保证实体字段、SQL字段、页面表单和权限校验一致，避免页面能提交但后端状态不可控。
4. 论坛的状态控制需要落到后端，包括文章锁定、分类归属、访问计数、管理员操作和权限判断；前端只负责表达操作入口，是否能发帖、编辑、删除、管理分类必须由后端角色权限决定。
### 03. BS用户管理系统需求分析文档.md
B/S用户管理系统不要只做几张登录注册页面，关键要把普通用户和管理员进入系统后的权限边界讲明白，注册、登录、资料修改、用户查询和删除都要落到可验证的账号状态上。
1. 登录页是系统入口，用户输入登录名和密码后，后端根据账号类型进入普通用户主页或管理员主页。登录失败时返回错误提示，登录成功后创建会话并控制后续页面访问范围。
2. 普通用户登录后只能维护自身信息，可以进入修改账户页面，修改密码、年龄、性别等资料，并通过注销返回登录页；注册和修改页面需要执行必填校验、密码长度校验、确认密码一致校验、年龄范围校验和性别值校验。
3. 管理员登录后进入用户管理工作台，可以查看所有注册用户，按查询条件检索用户，删除选中的用户，并通过注销退出系统。用户列表、查询页和删除页应共用同一套用户数据源，保证查询结果和删除结果实时一致。
4. 开发时应把账号、角色、会话、表单校验、错误提示和管理员操作权限统一封装。前端页面可以提供注册、查询、删除入口，但最终是否允许访问和修改必须由后端身份判断控制。
### 04. cms内容管理系统需求规约(新).md
做CMS时先别急着堆新闻列表，站点群、栏目树、内容库、模板发布和管理员授权才是主线；多个站点一起维护时，内容归属、发布状态和操作权限必须从后台就分清楚。
1. 站点群管理是系统顶层能力，平台支持主站和子站创建，子站可配置独立目录、独立域名或二级域名，并支持站点间内容转移、同步和分发。每个站点拥有自己的栏目、内容空间和管理员授权范围。
2. 栏目管理支持无限级分类，管理员可按业务需要创建、修改、删除栏目，并为不同栏目绑定模板、显示规则和权限。栏目既承载新闻，也可承载产品、文档、下载等不同内容类型。
3. 内容管理负责内容新增、编辑、分类发布、评论管理、调查投票和模板调用。系统管理员拥有用户、权限、栏目、模板、评论、投票和系统配置管理能力，一般管理员只能在授权站点或栏目下操作。
4. 开发实现应把站点、栏目、内容、模板、附件、评论、投票和管理员权限建成清晰模型。前端展示由模板和发布状态驱动，后台操作由角色和栏目授权约束，保证大型站点群下内容不串站、不越权、不重复维护。
### 05. ~$用户管理系统需求分析文档.md
这个用户管理系统可以按通用后台底座来写，先把账号注册、登录会话、个人资料维护和管理员集中处理拆开，再补上角色限制和操作留痕，避免用户越权改到别人的资料。
1. 用户注册时填写账号、密码和基础资料，后端完成必填校验、密码规则校验和账号唯一性校验，注册成功后进入登录流程。
2. 用户登录后进入个人中心，能够查看自身资料、修改密码和更新联系方式等信息。所有修改都需要后端重新校验字段合法性，避免前端绕过规则提交异常数据。
3. 管理员登录后进入用户管理列表，可查询用户、筛选用户、查看详情、删除用户或调整用户状态。管理员操作应写入日志，保证后续可审计、可追溯。
4. 系统实现上应把账号、角色、用户资料、登录会话和操作日志作为基础模型，后续可扩展到 RBAC 权限、组织机构和多管理员分级管理。
### 06. 个人网站需求分析.md
个人网站看起来简单，但需求要从访客第一次进入首页开始写清楚：个人资料、日志、相册、学习内容、音乐影视和留言板各有入口，后台维护也要能跟前台栏目对应起来。
1. 首页负责整体形象展示和栏目入口，需要突出个人主题、导航结构、最新日志、推荐相册、学习内容和联系方式，让访客能快速理解网站定位。
2. 个人信息模块展示站主基本资料、兴趣、经历和联系方式；日志模块用于发布文章、随笔和思想内容；相册模块展示个人照片和收藏图片；音乐影视模块承载兴趣内容；学习园地用于整理课程、资料和学习心得。
3. 留言板提供访客反馈入口，访客可以提交留言，站主可以查看、回复或删除不合适内容。留言功能应考虑基础防刷、敏感内容过滤和后台审核。
4. 开发上应把栏目、文章、图片、留言和媒体资源统一管理，保证前台展示简洁，后台维护方便。响应速度、页面风格和响应式适配是个人网站体验的关键。
### 07. 二手交易平台需求分析.md
落到校园二手交易这个场景，重点不是泛泛做商品展示，而是让学生把闲置物品发布出去、让买家能搜索沟通、让管理员能处理供求信息和订单异常，校园线下交易的特点也要保留下来。
1. 游客可以浏览商品分类、查看供求信息和搜索商品，但发布、留言、收藏和交易需要注册登录；用户注册后进入会员体系，能够维护个人资料、修改密码、找回密码，并保留自己的发布、收藏和订单数据。
2. 卖家在前端发布商品或供求信息，填写商品类别、标题、描述、价格、联系方式和图片等内容，后端生成商品记录并进入展示列表。卖家可以管理自己发布的商品，修改商品信息、删除下架商品、查看留言并回复买家咨询。
3. 买家通过分类、搜索、商品详情和公告了解商品，感兴趣后可留言、收藏或发起购买。订单产生后，买家可以查看订单状态、取消订单、确认收货；卖家可以查看订单并处理交易沟通。
4. 管理员通过后台维护公告、用户、供求信息、商品分类、留言和订单数据。平台交易以校园熟人环境和线下确认优势为基础，一期可以先完成信息撮合和订单记录，支付接口可作为后续能力接入。
### 08. 太空射击网页游戏与排行榜系统需求文档.md
太空射击项目适合改成可运行的网页游戏全栈项目，前端负责飞船操控、敌机生成、碰撞和关卡动画，后端负责玩家账号、得分提交、排行榜校验和历史记录。
1. 开始界面提供开始、退出和排行榜入口；玩家点击开始后载入第一关，界面展示玩家生命值和当前得分；点击排行榜可查看历史玩家分数，点击退出则释放资源并结束游戏。
2. 游戏过程中敌方目标持续向玩家飞船移动，目标与飞船碰撞后玩家生命值减少。玩家发射子弹击中目标后，目标销毁并增加积分，单次命中奖励按规则累加。
3. 关卡推进由分数驱动，积分达到4000进入第二关，达到8000进入第三关，并按相同逻辑继续推进。玩家生命耗尽进入失败界面，通关进入胜利界面。
4. 胜利或失败后，玩家输入姓名，系统将姓名和得分写入数据库，再跳转排行榜展示名次。开发上应围绕场景切换、碰撞检测、子弹命中、生命值、得分阈值和排行榜持久化组织代码。
### 09. 会展管理系统需求文档.md
会展现场最怕人工登记混乱，所以这套系统要先围绕条码和手持设备设计：入场、餐饮、图册发放、部门时段和统计报表都从一次扫码动作延伸出去。
1. 系统管理模块负责用户设置、参数配置和基础权限；条码信息管理负责录入参会人员或参展对象的条码数据；手持机管理支持设备新增、删除、修改和现场使用配置。
2. 部门管理和时段设置用于组织会展现场规则，管理员可维护部门信息、配置多时段入场或服务时间，使扫描核验能够按部门、时段和条码权限执行。
3. 扫描入场模块通过手持设备读取条码，后端校验条码有效性并保存入场记录；餐饮消费模块扫描条码后验证消费资格、扣除餐费并记录消费数据；图册发放模块扫描条码后保存发放记录并统计发放数量。
4. 统计报表模块汇总入场、餐饮、图册发放、部门和时段数据。开发时应保证离线采集、重复扫描控制、设备同步、条码权限和报表口径一致。
### 10. 公司企业网站需求文档.md
企业网站不能只停留在公司介绍页，产品展示、新闻动态、下载中心、客户留言、招聘和后台栏目维护都要安排清楚，让运营人员后续可以持续更新内容。
1. 前台栏目包括企业介绍、信誉认证、组织结构、产品展示、产品搜索、新闻中心、下载中心、客户服务、BBS论坛、人力资源、合作链接、营销中心、网站导航和联系方式。首页需要集中呈现公司形象、产品入口、新闻入口、会员注册、站内搜索和客户留言。
2. 产品展示系统按产品系列和多层分类组织内容，支持图片、文字、分类检索和关键词搜索；管理员可以在后台维护产品分类、产品详情、推荐产品和产品资料，使访客能够按目的快速找到产品。
3. 新闻发布系统用于发布公司动态、行业信息、媒体报道和合作交流内容，支持按时间排序、人工置顶、推荐、关键词查询和前台展示，下载中心负责软件、图片、文件等资料分类管理和上传维护。
4. 会员和客户服务模块支持客户注册、会员类别、权限范围、审核、留言、分类管理和回复。招聘系统支持职位发布、简历提交、简历入库、统计、检索和分析。
5. 后台是整个网站的运营中枢，需要支持多人、多层次权限，不同栏目或版块可由不同管理员维护。所有前台数据都从后台生成，开发时要保证内容、权限、统计和页面展示统一。
### 11. 公文管理需求分析文档.md
公文管理要从一份公文被创建的那一刻写起，呈报、签发、退回、跟踪、待办、查询和归档都属于同一条生命周期，不能让流程状态只停留在页面说明里。
1. 公文呈报模块负责发起公文，用户填写公文标题、正文、类别、密级、接收对象等信息后提交，系统生成公文记录并进入流转流程。
2. 公文签发模块为签发人生成待签发列表，签发人进入待办任务后查看公文内容并执行签发、退回或补充意见等操作。办理完成后，系统更新公文流转状态。
3. 公文跟踪和流程监控模块用于查看公文当前所处环节、已批环节、审批内容和处理人，让发起人和管理人员能够掌握流转进度。
4. 查询模块分发文查询、收文查询、确切查询和模糊查询；归档查询负责历史发文和收文检索；待办公文模块以列表方式展示当前用户需要办理的事项，处理完成后自动移出待办。
5. 开发时要把公文、流程节点、处理人、审批意见、待办状态、归档状态和权限范围统一建模，保证用户只能查看自己呈报、签发或待办范围内的公文。
### 12. 名片管理系统需求文档.md
名片管理的价值在于把零散联系人变成可维护资产，录入名片时就要考虑同一人物、多家公司、部门职务、分组查询和访问记录这些后续使用场景。
1. 用户管理模块支持登录、找回密码、修改密码和一周内免登录。普通用户进入系统后维护自己的名片数据，超级管理员负责审核用户申请、维护用户和权限。
2. 名片管理模块支持名片一览、详情查看、更新、删除、批量删除、生成名片、同一人物设定、公司部门视图和访问日志，系统需要识别同一人物的多张名片，把个人、公司、部门、职务和联系方式整理成可查询资料。
3. 组管理模块支持新增组、查询组、向组内添加名片、查看组详情、修改组、删除组和查看组内名片。分组用于按业务、客户、项目或部门组织联系人。
4. 参数管理模块负责用户信息、权限、公司信息、企业信息、部门信息和职务信息维护。开发时应把名片、联系人、公司、部门、职务、分组和访问日志统一建模，支持后续 CRM 化扩展。
### 13. 搜索引擎系统需求分析说明书.md
简易搜索引擎可以从“用户指定一个目录”开始拆需求，索引生成、关键字匹配、结果排序、文件信息展示和索引更新都要讲清楚，避免最后只剩一个输入框。
1. 索引生成模块负责读取用户指定文件夹，对目录内文件建立索引数据。索引内容可以包含文件名、路径、类型、关键内容和更新时间，为后续搜索提供基础。
2. 关键字查询模块接收用户输入的搜索词，后端基于索引进行匹配，返回符合条件的文件列表。结果页需要展示文件名、路径、匹配信息和可打开入口。
3. 系统用例以生成索引和利用关键字查找文件为核心，用户必须先完成索引生成，才能获得稳定的搜索结果。若文件夹发生变化，需要支持重新生成或更新索引。
4. 开发时要处理索引存储、关键字匹配、文件类型兼容、搜索性能、异常文件跳过和结果排序。该系统可扩展为站内搜索、资料库搜索或轻量文档检索工具。
### 14. 火车购票系统需求分析.md
火车购票系统先抓住车次和余票这两个对象，用户查询、旅客订票、售票员确认、退票回库和座位释放都要共享同一套库存变化规则。
1. 车票记录模块展示数据库中所有车票和车次记录，包括车次、区间、发车时刻、到站时刻、座位和余票信息。用户可以按车次查询，也可以按出发站和到达站进行区间查询。
2. 查询结果需要支持排序，尤其是按发车时刻和到站时刻排序。车次基础信息面向普通用户只提供查询，不允许随意修改，后台或售票员角色负责维护车次和票务数据。
3. 客户根据查询结果选择车次并输入个人信息完成订票，系统生成订票记录。售票员可根据旅客指定车次查询可用座位，或根据站点查询经过车站的所有车次，再完成窗口售票。
4. 退票模块允许客户按姓名或订票信息查找订单并申请退票，系统办理退票后将车票重新释放回库存，使退回车票可以继续出售。开发时要重点控制余票扣减、退票回补、并发购票和订单状态一致。
### 15. 系部事务管理需求分析文档.md

### 27. bbs需求文档.md
企业内部BBS的重点是让员工发信息、看讨论、交流岗位内容，同时让版主和管理员能处理版块、主题、回复和权限，B/S架构只是承载方式。

### 28. 大件订单多维度履约调度系统需求文档.md

5. 待调度管理处理人员登出、过期和非操作状态下的订单释放。调度报表统计调度订单数据、人员处理时效和恶意拒收线索，并可向风控输出风险客户。

4. 工资模块根据基本工资、补贴、公积金、养老金、所得税、考勤和表现生成月工资，支持个人工资报表和部门工资报表。
5. 查询统计模块支持按员工号、姓名、部门、工资、培训、考勤等条件查询，并将符合条件的记录以报表形式打印。开发时应保证员工历史记录、部门关系和薪资计算可追溯。

### 31. 任务管理系统需求分析.md
任务管理要从主管创建任务开始，分派给员工后还要看到计划拆解、执行反馈、状态变化和最终确认，不能只做一个任务列表。
1. 管理员负责创建系统用户并分配角色，角色包括主管和员工。主管和员工进入系统后看到不同功能入口。
2. 主管制定任务时填写任务名称、描述、起止时间和实施人，一项任务只能分配给一名员工。任务创建成功后进入待实施状态。
3. 主管可以查看自己创建的任务列表，并按实施人、任务状态、开始时间和结束时间排序。任务详情展示任务信息和该任务下的计划信息。
4. 主管可调整未实施任务，修改任务名称、时间、描述和实施人，也可删除未实施任务；实施中和已完成任务的修改受限制。
5. 员工收到任务后制定计划，计划包括计划名称、起止时间、计划描述和状态。计划完成后员工填写反馈信息并标记已反馈，主管查看反馈后确认任务是否完成。

### 35. 大转盘和砸蛋需求文档.md
大转盘和砸蛋活动不要只写成前台动画，后台配置、积分扣减、概率控制、奖品库存、中奖记录和领取状态才是营销活动能不能跑稳的关键。
1. 大转盘活动由后台配置起止时间、消耗积分、奖品档位、中奖概率、机会次数和奖品图片链接。前台展示活动时间、倒计时、奖品、参与人数、中奖公告和中奖人数。
2. 用户点击立即开始后，系统先校验积分是否足够，足够则扣除积分并启动转盘，不足则弹窗提示并引导到获取积分说明页。
3. 转盘中奖后弹窗展示奖品名称，用户可继续使用剩余机会；未中奖则提示未中奖状态。我的奖品展示中奖记录、领奖单和已领取状态。
4. 砸蛋活动同样支持后台配置消耗积分、每局机会、奖品、中奖概率和奖励积分。用户中奖后可继续玩剩余次数，也可立即兑换，奖励积分进入账户。
5. 开发要保证积分扣减、概率计算、奖品库存、领取状态、中奖名单和防重复提交稳定一致。

### 41. 货币基金理财账户系统需求文档【3】.md
货币基金理财账户系统要按资金流写清楚，基金份额、账户余额、消费抵扣、实时赎回、普通提现、正常赎回和收益报表都要保证金额口径一致。
1. 支付场景中，用户在收银台选择商品或服务并下单。若货币基金可用金额大于0，系统在账户余额下展示可用金额。用户选择货币基金支付后，浮层展示全部可用金额和本次最多可用金额。
2. 用户输入支付密码后，系统实时扣除指定基金份额，同步更新用户资产可用金额，并异步通知基金公司。扣除失败时展示失败原因；退款时货币基金支付金额统一退回账户余额。
3. 实时赎回到余额流程中，用户输入赎回金额，系统校验最高可赎回金额、单笔额度、日累计额度和月累计额度，成功后扣除基金份额并转入账户余额。
4. 普通提现先把货币基金实时赎回到账户余额，再按普通提现规则提现到银行卡；正常赎回先赎回到理财账户余额，再代发到银行卡，用户可获得当日收益。
5. 资产管理页面展示可用金额、累计收益、快速购买、定投设置、实时赎回、转入明细和转出明细。后台报表统计购买、消费、赎回、提现、失败订单和定投失败等指标。

### 43. 日志管理系统需求分析.md
日志管理系统要从员工每天填报开始写，日报、项目反馈、主管评价、经理查看和缺失提醒都要能沉淀成后续统计依据。
1. 员工登录后可以新增、编辑、删除自己的工作日志，查看每天、每周、每月日志，标记计划是否完成，并记录提醒、周期重复和相关费用。

3. 部门经理可以查看部门职员日志、评价日志、查看项目反馈，并把需要上报的信息转发给总经理。
4. 总经理可以查询所有员工日志和项目情况，按人、时间、项目维度查看历史记录和汇总统计报表。
5. 系统管理员负责账号、角色和错误数据维护。系统应自动识别未按时提交日志或项目反馈的员工，并生成缺失名单发送给部门经理。

### 45. 权限管理详细需求文档.md
权限管理系统可以从RBAC模型落地讲起，用户、角色、菜单、页面按钮和资源路径之间要有明确授权关系，前端入口和后端校验必须一致。
1. 系统默认root用户拥有root角色和root权限，root权限不在页面展示。普通用户可以拥有多个角色，角色可以勾选多个具体权限。
2. 终端管理权限包括终端查看、终端维护和终端组全见。维护和全见权限依赖查看权限，勾选高级权限时应自动具备基础查看能力。
3. 素材管理权限包括素材查看、审核、上传、删除、全见和组内共享。布局管理权限包括布局查看、管理、全见和组内共享。
4. 节目管理权限包括查看预览、制作、修改、删除、全见和组内共享；播放计划权限包括查看、制作、修改、删除、审核、全见和组内共享。
5. 用户管理、日志管理和系统管理分别控制用户/用户组、角色、日志查看导出、系统设置、FTP服务器、布局分辨率和在线用户管理。开发时前后端都必须校验权限，不能只靠菜单隐藏。
### 46. 社区联盟公会系统需求文档.md
猫扑联盟更像社区里的公会系统，创建联盟、加入成员、贡献声望、等级成长、排行榜奖励和小窝互动要共同推动用户活跃。

2. 加入和退出流程受规则控制，盟主不能直接退出，需要移交盟主、被弹劾或系统修改后才能退出。成员退出或被开除会扣减联盟声望，并清空成员贡献。
3. 联盟声望由成员发帖、回帖、登录、成员加入等行为增加，也会因成员退出、开除、删帖、删回复和等级维护消耗。声望不足时联盟降级，一级联盟声望归零可解散。
4. 用户贡献度由成员对联盟声望增长的贡献计算，贡献度用于成员排行和奖励分配；联盟等级、人数、每日消耗和奖励规则共同形成成长体系。
5. 我的联盟、别人的联盟单页、我的联盟单页和联盟小窝承载成员列表、联盟信息、社交互动和二期扩展。开发时要以声望流水和贡献流水保证规则可追溯。
### 47. 用户权限管理系统需求分析.md
通用用户权限管理要考虑被多个业务系统复用，登录安全、用户维护、角色授权、资源控制和操作审计都要抽成稳定的授权底座。
1. 登录流程先判断用户IP是否在黑名单内，再校验用户名和密码。登录成功后，系统根据用户权限初始化对应工作窗口。
2. 用户管理负责用户信息录入、维护、授权和组织机构分配，支持按部门编号、用户编号和用户姓名检索。
3. 角色管理把权限相似的用户归类，角色可分系统管理员、管理员、用户、访客等，并支持上下级关系。下级角色授权范围不能超过上级角色。
4. 组织管理用于按企业部门或机构组织用户，只提供组织视图和分组，不直接参与权限计算。
5. 资源管理控制页面、图片、附件等路径访问，操作管理控制按钮和动作权限。日志管理记录权限相关行为，IP黑名单提供第一道访问防护。

1. 系统登录、工作台和日志查询构成用户进入后台后的基础工作区，所有关键操作需要记录日志，支持后续审计。
2. 系统管理维护机构、角色、部门、人员和FA贷款合作商。不同岗位根据权限进入不同贷款环节和管理模块。
3. 贷款管理包括全部贷款、待分配、初审、尽职调研、复审、授信、实时风险预警、贷款参数、电子合同和债权打包。贷款在各阶段流转时，列表需要支持客户、证件号码、所属FA、所属机构等条件查询。
4. 金融机构管理维护金融机构列表和金融产品，复审通过时可选择金融产品；客户管理维护客户列表、贷款支付订单、客户信用记录和手动信用记录。
5. 黑名单管理支持加入和取消黑名单，内容管理支持公告、内容和分类。开发时要重点保证流程状态、审批权限、合同、债权、风险预警、黑名单和数据安全。
### 59. 超市管理系统需求文档.md
超市管理系统要把前台收银和后台库存放在一起看，销售发生后库存变化、会员客户、进退货、人员权限和经营报表都要同步更新。
1. 用户角色包括销售员、仓库管理员、普通管理员和超级管理员。销售员只能进入销售界面，仓库管理员只能进入仓库管理，普通管理员进入部分人员管理，超级管理员可访问全部功能并管理权限。
2. 销售模块通过商品二维码和数量完成销售，系统判断批发或零售，并根据库存临界值提示是否需要进货。
3. 人员管理模块维护会员、客户、员工和管理员信息，支持新增、查询、修改、删除和权限调整。
4. 仓库管理模块支持库存查询、商品增删改查、进货记录、出库记录、退货记录、报损报溢统计和库存状态维护。
5. 综合统计和报表模块统计销售总额、财务报表、退货报表、出库报表、入库报表和销售报表。开发时要保证销售、库存和财务数据实时一致。
### 60. 通用网上商城需求文档.md
通用网上商城可以按一笔订单从产生到关闭来写，顾客注册选品、加入购物车、提交订单、审核付款、发货关闭和后台分工都要落到订单状态里。
1. 顾客注册时填写用户名、密码和邮箱，注册后可补充个人资料和收货人信息，后续可修改密码、查询订单和修改未处理订单。
2. 选品流程支持顾客在首页、专柜页、商品分类页、搜索结果页和详情页把商品加入购物车。购物车可修改数量、取消某商品或清空。
3. 订单提交时，系统读取或要求填写收货人信息，生成订单并进入“提交成功、尚未审核”状态。管理员审核有效后变为“已审核、尚未付款”，信息无效则变为“无效订单”。
4. 顾客付款后订单进入发货流程，后续状态变为已发货和订单关闭；一定时间内未付款的订单可自动取消，状态变化可邮件通知用户。
5. 后台分超级管理员、订单管理员和界面管理员。超级管理员拥有最高权限，订单管理员负责订单状态和会员信息，界面管理员负责商品和广告维护。
### 61. 银行系统需求分析.md
银行储蓄系统必须先保证账户和资金一致，开户销户、存款取款、查询转账、密码维护和交易流水都要受身份校验与事务控制约束。
1. 登录模块根据不同身份进入不同界面，储户和管理员看到不同操作范围。
2. 开户和销户模块处理账户生命周期；储户提交销户申请后，由管理员执行销户操作，系统更新或清除储户相关信息。
3. 存款和取款模块每次交易都写入数据库，交易完成后账户余额必须同步更新，并保留交易记录。
4. 查询模块为储户提供余额查询和交易记录查询，转账模块处理账户间资金变动，密码修改模块允许用户维护账户安全。
5. 系统必须保证金额、身份证、卡号等数据精确，数据库操作前后保持同步。开发时要重点处理事务一致性、并发交易、异常回滚、密码安全和审计日志。
