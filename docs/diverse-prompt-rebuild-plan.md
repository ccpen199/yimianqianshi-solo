# Diverse Prompt Rebuild Plan

## Goal

保留已完成项目，删除并重建所有未完成项目。新数据必须减少重复产品名、重复流程和重复功能组合，避免继续沿用“用户 + 场景 + 界面/看板”的旧结构。

## Rebuild Strategy

新数据从原始 Chrome Web Store 分类素材出发，只借参考产品的方向和能力，不复制品牌或页面。

每个新项目由以下维度组合：

- 业务域：购物、娱乐、社交、旅行、游戏、健康、家居、新闻天气、设计、休闲。
- 目标人群：按业务域重写，不再统一使用泛化“用户”。
- 真实场景：比如优惠核销、签证核验、MOD 冲突、维修报价、素材授权。
- 业务事故：比如跨站价格漂移、航班延误重排、粉丝标签漂移、素材授权缺失。
- 数据条件：比如脏数据、缺字段、重复记录、限流接口、缓存旧值、时区偏差。
- 产品形态：规则引擎、复核台、调度器、证据包、校验器、协作流。
- 主流程：录入或导入对象 -> 校验关键字段 -> 识别异常与冲突 -> 给出处理建议 -> 人工确认或回滚 -> 导出结果与留痕。

## Anti-Repetition Rules

- 不再用旧的 `界面应用 / 全栈应用 / 数据看板` 作为产品形态。
- 未完成项目名称中禁止出现 `数据看板 / 看板 / 界面 / 界面应用 / 全栈应用`。
- 产品名由多模板生成，混合微场景、数据条件、业务事故、能力和产品形态。
- 名称去重不能靠数字尾巴，必须换业务词。
- 每条 prompt 都包含正常、边界、冲突、失败四组 mock 数据要求。
